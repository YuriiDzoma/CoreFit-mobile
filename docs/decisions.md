# Architecture Decisions

Lightweight decision log for the choices that shape CoreFit Mobile's foundation. Each entry is Context (the problem), Decision (what we chose), Consequences (what that implies going forward).

## Expo Router

**Context:** Need a routing solution for a universal (iOS/Android/web) Expo app that scales past a handful of screens without hand-wired navigation config.

**Decision:** File-based routing via `expo-router`, with routes under `src/app` and `experiments.typedRoutes` enabled in `app.config.ts`.

**Consequences:** New screens are added by creating files, not by registering them in a navigator config. Typed routes catch invalid `href`s at compile time. Route groups and layouts (`_layout.tsx`) are the mechanism for shared chrome (tabs, headers).

## Supabase

**Context:** Need auth, a relational database, and file storage without standing up and operating custom backend infrastructure.

**Decision:** Supabase as the backend ŌĆö Postgres database, built-in auth, storage, and row-level security (RLS) for authorization. Considered Firebase briefly; Supabase was chosen for SQL/Postgres (easier relational modeling for workout/progress data) and a first-class TypeScript client.

**Consequences:** Client code talks to Supabase directly from the app via its JS client (no custom API server layer to start). Authorization logic lives in Postgres RLS policies, not application code. `EXPO_PUBLIC_SUPABASE_URL` and `EXPO_PUBLIC_SUPABASE_ANON_KEY` (see `.env.example`) are the only credentials the client needs ŌĆö the anon key is safe to embed in the bundle by design.

## TanStack Query

**Context:** Screens will need to fetch, cache, and revalidate data from Supabase without each screen re-implementing loading/error/refetch state.

**Decision:** `@tanstack/react-query` as the server-state layer, wrapping Supabase calls in query/mutation hooks.

**Consequences:** Server data is cached and deduplicated across screens automatically. Loading/error UI follows one consistent pattern. Server state (from Supabase) and client state (UI-only) stay clearly separated ŌĆö server state does not belong in Zustand.

## Profile data layer (Foundation)

**Context:** The app needs a typed, validated way to read and update the `profiles` table before any profile UI is built. The `profiles` schema isn't checked into either this repo or the web repo as migrations/DDL ŌĆö the only authoritative source is the live Supabase project. `profiles` is also the first custom (non-auth) table the mobile app talks to, so this is the first real decision point for how table access should be structured, ahead of TanStack Query landing.

**Decision:** `src/lib/supabase/profile.ts` is the sole place that touches `supabase.from('profiles')`, mirroring `src/lib/supabase/auth.ts`'s existing convention exactly ŌĆö no separate repository/service split, one file, plain Promise-based functions (`getProfileById`, `updateProfileById`). The `profiles` schema is defined as a Zod object matching the live database (columns, types, and nullability confirmed directly against the Supabase project via `supabase db query`, not inferred from either app's client code), and every Supabase response is parsed through it before being returned. `ProfileUpdate` is scoped to only the columns that are actually meant to be user-editable (`username`, `avatar_url`, `language`, `dark`) ŌĆö `email` and `is_trainer` are excluded even though the table's RLS `UPDATE` policy doesn't itself restrict them, because neither is meant to be changed this way (`email` belongs to the auth flow; `is_trainer` is backend/admin-managed).

No `useProfile` hook or any client-side cache/store is introduced yet ŌĆö this layer is deliberately Promise-only. Hooks and caching are deferred until TanStack Query is introduced (see above), once there are enough server-state features to justify it; adding a throwaway loading hook now would just be removed and replaced later.

**Consequences:** Any future table beyond `profiles` should default to this same single-file convention (schema + types + thin functions, colocated) unless it grows enough distinct concerns to justify splitting. When TanStack Query lands, `profile.ts`'s functions become the query/mutation functions passed into hooks ŌĆö nothing about this layer's shape needs to change, only a hook layer gets added on top. Because RLS on `profiles` allows public `SELECT` access (including unauthenticated reads, and including `email`), callers of `getProfileById` must not assume the requester is viewing their own data ŌĆö this file intentionally does not special-case "own profile" fetching.

## Exercise data layer and client-side filtering

**Context:** The web app's exercise "wiki" (`exercises` + `muscle_groups` tables, confirmed live: 61 exercises, 7 muscle groups, RLS disabled on both) issues a fresh Supabase query every time the user taps a muscle-group filter tab, plus a separate round-trip to resolve the tapped tab's hardcoded English name to a `muscle_group_id` before it can even run that query. With only 61 rows total, the entire table is cheap to hold in memory at once, and mobile's first exercise screen (`src/app/(app)/explore.tsx`) needed a filtering approach to decide on before this could be built at all.

**Decision:** `src/lib/supabase/exercises.ts` fetches the full `exercises` and `muscle_groups` tables once, unfiltered (`getExercises()` takes no parameters), mirroring `profile.ts`'s single-file convention and colocating the raw Zod schema/types with a `localizeExercise` mapping function in the same file rather than a separate module. Muscle-group filtering (`selectedMuscleGroup: string | null`, where `null` means "All") and the eventual name/description text search are both applied client-side, in-memory, in the screen itself ŌĆö not as Supabase query parameters, and "All" is never written to or read from the database as a real `muscle_groups` row, it's purely a UI-level `null`. `ExerciseRow` (the raw per-language DB shape) and `LocalizedExercise` (the UI-facing, single-language-resolved shape produced by `localizeExercise`) are kept as two distinct types on purpose, even though only the `'en'` locale is used anywhere today ŌĆö this is the one seam where real i18n would plug in later without changing the data layer or the fetch functions.

**Consequences:** Selecting a filter tab is instant and offline-safe (no network dependency once the initial fetch completes), and adding a text-search box later is free ŌĆö it's just another client-side predicate over the same in-memory list, no new query shape needed. This approach only works because the dataset is small (tens of rows, not thousands); it should be revisited (server-side filtering, pagination) if the exercise catalog grows enough that fetching it whole becomes expensive ŌĆö not applied preemptively while the current numbers make it unnecessary. Exercise cards in the list are already `Pressable` (initially with a placeholder handler, now wired to the detail screen ŌĆö see below); video (`video_url`) is deliberately not rendered or fetched-for-display anywhere yet ŌĆö it's carried through `ExerciseRow`/`LocalizedExercise` for completeness but has no consumer until its video-embedding approach (likely `react-native-webview`, not yet added) is built.

## Exercise detail screen and nested Stack inside a native tab

**Context:** `explore` (one of three `NativeTabs` ŌĆö `expo-router/unstable-native-tabs`, the experimental native-tab-bar API) was a flat file with no push navigation. `NativeTabs` has no JS mock header the way `Tabs` does, so there was no built-in way to push a detail screen or show a back affordance within that tab without extra structure. Separately, `getExerciseById`'s `.single()` throws Supabase's generic `PGRST116` error on a missing row, and the exercise `type` column, while currently constrained by a DB `CHECK` to `'compound'`/`'isolation'`, isn't guaranteed to stay that way from the client's perspective.

**Decision:** `explore.tsx` became `explore/` with a nested `Stack` (`_layout.tsx`, `headerShown: false` to match every other `Stack` in this app) wrapping `index.tsx` (the list, unchanged aside from `handleExercisePress` now calling `router.push`) and `[id].tsx` (new). This is Expo's documented pattern for `NativeTabs` when a tab needs push navigation. The detail screen keeps the app's existing manual-back-link convention (`<Link href="/explore">`, same shape as "Back to sign in" in the auth screens) rather than turning on a native header, for visual consistency with the rest of the app rather than introducing native header chrome in one isolated place. `getExerciseById`'s `.single()` `PGRST116` error is caught specifically and mapped to a distinct not-found state, separate from generic errors ŌĆö web's equivalent page has no such handling at all. `exerciseRowSchema.type` was loosened from a strict Zod enum to a plain nullable string, and a local `formatExerciseType` in `[id].tsx` maps known values to labels with an explicit default case for anything else ŌĆö an enum at the schema layer would make the _entire_ fetch throw on an unexpected value, which is the opposite of the graceful handling wanted for that field.

**Consequences:** Any future tab needing push navigation (e.g. a training/programs tab) should follow the same nested-`Stack`-with-`headerShown: false` shape rather than reaching for a native header by default. The detail screen independently re-fetches by id rather than reusing the list's in-memory data (matching web's own behavior) ŌĆö a deliberate, accepted redundancy at this data size (61 rows), not an oversight; revisit only if a cache layer (TanStack Query) is introduced for other reasons. React Query itself remains deliberately deferred ŌĆö this is now the third server-state feature (profiles, exercise list, exercise detail) without one, a conscious choice to keep re-evaluating rather than a default.

## Exercise video via YouTube embed

**Context:** `exercises.video_url` is always a YouTube URL, but not a uniform shape ŌĆö checked live: 57/61 rows are `watch?v={id}`, 4/61 are `shorts/{id}`. Web's detail page embeds video with `data.video.replace("watch?v=", "embed/")`, which does nothing on the 4 `shorts/` rows, leaving a non-embeddable URL as the iframe `src` ŌĆö a real bug in web, confirmed against actual data, not a hypothetical. `expo-video` can't play YouTube URLs at all (no direct file URL YouTube exposes publicly), so an embedded web view is the only viable approach on mobile too, same underlying reason web uses an iframe instead of `<video>`.

**Decision:** `src/lib/youtube-url.ts` exports `getYouTubeEmbedUrl(url): string | null`, using the `URL` API (hostname, pathname, `searchParams`) rather than a single regex ŌĆö each supported shape (`watch?v=`, `youtu.be/`, `shorts/`, and already-`embed/`-form URLs) is parsed structurally and always rebuilt into a canonical `https://www.youtube.com/embed/{id}`, fixing the `shorts/` case rather than porting web's bug. `null` in, or anything unparseable/unsupported, returns `null` ŌĆö no partial/broken URLs ever reach the player. `src/components/youtube-embed.tsx` (native, `react-native-webview`) and `.web.tsx` (a plain `<iframe>`, matching the existing platform-split convention used by `app-tabs.web.tsx`/`animated-icon.web.tsx`) both take only `{ url, title }` ŌĆö all normalization and the "should anything render" decision stay inside the component, so the detail screen's call site never has to know about embed URLs or YouTube's URL shapes at all. `react-native-webview` needed no config plugin; since this project uses a custom dev client (not bare Expo Go), installing it requires a native rebuild before it's testable.

**Consequences:** Any other embedded third-party player this app adds later (unlikely soon, but the shape is now established) should follow the same self-contained-component pattern ŌĆö normalization logic lives with the embed, never leaks into call sites. `react-native-webview` has no web-platform support itself, which is exactly why the `.web.tsx` split exists rather than trying to make one component branch internally on `Platform.OS`. No autoplay, no play-state tracking ŌĆö matches web's actual behavior, not an arbitrary restriction.

## Programs Foundation: dedicated tab, Home reserved for the future Dashboard

**Context:** Training Programs is the next web feature being ported, chosen because web's own exercise-picker component reuses the exercise wiki's data layer and UI directly, and because Complexes/workout-logging/the home feed are all downstream of Programs existing first. The `programs` table has two distinct owner-ish columns ŌĆö `user_id` (nullable, FK to `auth.users`, what web's own list query filters on) and `author_id` (NOT NULL, FK to `profiles`, used for author-join purposes elsewhere) ŌĆö easy to conflate, so worth naming explicitly here: the list filters on `user_id`, matching web exactly.

**Decision:** A new, dedicated "Programs" tab (`src/app/(app)/programs/`) rather than repurposing the still-boilerplate Home tab ŌĆö Home stays untouched, reserved for a future Dashboard once Programs, workout logging, and training history exist to feed it (matching web, where the home feed is itself downstream of those same features). The route is folder-first from day one (`programs/_layout.tsx` + `programs/index.tsx`) rather than starting as a flat file the way `explore.tsx` did ŌĆö avoiding the flat-file-to-folder migration Explore needed between Sprints 12 and 13 once its detail screen landed. `getPrograms(userId)` (not `getUserPrograms`) to keep the data-layer naming consistent with future sibling functions (`getProgram()`, `getGlobalPrograms()`, etc.). The list-row presentation is a standalone `ProgramCard` component (not inlined in the screen) since web itself reuses the same program-item presentation in more than one place (Programs list, Global Programs) ŌĆö real, cited reuse, not speculative extraction. `type`/`level` are loosened from the DB's `CHECK`-constrained values to plain strings in the schema, with `ProgramCard` formatting known values and defaulting gracefully otherwise ŌĆö same reasoning as `exercises.type` in the previous entry.

**Consequences:** Any future tab needing push navigation should default to being created folder-first (`_layout.tsx` + `index.tsx` from the start) rather than repeating Explore's later migration, now that the pattern has been hit twice. The list's "+ Create Program" CTA and each `ProgramCard`'s `onPress` are already wired to placeholder handlers ŌĆö the create-wizard and detail-screen sprints are pure wiring, not a restructure, matching the exact precedent set by the exercise list before its detail screen existed. Home remaining reserved means there is currently no "landing" content in the app beyond the unmodified Expo Router starter template ŌĆö that's an accepted, deliberate gap, not an oversight, until the Dashboard's actual dependencies (Programs, logging, history) exist.

## Program Detail: nested query, read-only scope, shared helpers

**Context:** Web's `fetchProgramDetail` loads a program's full tree (days, exercises) in one query with nested PostgREST selects, then a separate call resolves exercise names by id. The same page also embeds workout-logging inputs, read-only training history, edit, and delete ŌĆö all of which depend on tables/features mobile hasn't built yet (`exercise_drafts`/`exercise_logs`/`training_history`, the creation wizard). Once a second screen needed program type/level formatting and not-found-vs-error detection, `formatProgramType`/`formatProgramLevel` (previously local to `ProgramCard`) and `isNotFoundError` (previously local to `explore/[id].tsx`) became genuine duplicates, not speculative extractions.

**Decision:** `getProgramDetail(id)` in `src/lib/supabase/programs.ts` uses one nested-select query (`programs ŌåÆ program_days ŌåÆ program_exercises`) ŌĆö the first nested/embedded Postgrest query in this codebase; every other table so far has used flat single-table selects, justified here because a program is genuinely a tree, not a justification for nesting elsewhere by default. `program_days` and `program_exercises` are sorted client-side by `day_number`/`order_index` after the fetch, since embedded relations aren't guaranteed to come back in a deterministic order ŌĆö a small correctness addition beyond what web itself does (web never explicitly sorts days). Exercise names are resolved via the existing `getExercises()` (already fetches the whole small table) rather than a new by-ids fetch function. `formatProgramType`/`formatProgramLevel` moved from `ProgramCard` into `programs.ts` (now used by both `ProgramCard` and the detail screen), and `isNotFoundError` moved from `explore/[id].tsx` into a new shared `src/lib/supabase/errors.ts` (now used by both detail screens) ŌĆö both moves triggered by actual duplication appearing, not preemptive extraction. Scope for this sprint is deliberately read-only: title/type/level and the day ŌåÆ exercise list only ŌĆö no training history, no logging inputs, no edit, no delete, no author link (mobile has no `/profile/[id]` route to link to), no density-mode toggle (web's own modes II/III are visually near-identical, so porting the 3-way toggle wasn't worth it for a first pass). No ownership check either ŌĆö nothing on this screen is ownership-gated yet, unlike web's `isMyProgram` gate, which only matters once edit/delete/logging exist.

**Consequences:** Any future table with a genuine tree/hierarchy shape (not just a one-to-many list) is a reasonable candidate for the same nested-select pattern; flat tables should keep using flat selects. `isNotFoundError`/`errors.ts` is now the place any future detail screen's `.single()` not-found handling should import from, not reimplement. This sprint deliberately doesn't distinguish "viewing your own program" from "viewing someone else's" ŌĆö that distinction was introduced in Sprint 21 alongside Delete (see "Program Deletion" below); Edit and logging still need to add their own owner-only affordances on top of the same check when they land.

## Zustand

**Context:** Need a place for client-only/UI state (e.g. active tab, in-progress form drafts, ephemeral UI flags) that doesn't belong in the server-state cache.

**Decision:** Zustand for client state ŌĆö minimal API, no boilerplate, no providers required.

**Consequences:** State that mirrors the backend goes through TanStack Query, not Zustand. Zustand stores stay small and UI-focused; if a store starts caching server data, that's a sign it should be a query instead.

## React Hook Form

**Context:** Auth, onboarding, and profile screens need validated forms with reasonable performance (no full-tree re-render per keystroke).

**Decision:** `react-hook-form` for form state and validation wiring, paired with Zod resolvers.

**Consequences:** Form validation schemas are defined once in Zod and reused for the resolver ŌĆö no separate hand-written validation logic per form.

## Zod

**Context:** Validation is needed in three places: parsing environment variables, validating form input, and validating/typing Supabase responses. Using a different tool for each would triplicate effort.

**Decision:** Zod as the single schema/validation library across all three use cases.

**Consequences:** One schema can, in principle, back both a form's validation and the shape of the data it submits to Supabase. Environment variable parsing (when added) uses Zod schemas rather than manual `if` checks.

## Password recovery as a dedicated auth state

**Context:** Supabase's password-recovery flow (`resetPasswordForEmail` ŌåÆ emailed link ŌåÆ `exchangeCodeForSession`) establishes a real session, indistinguishable from a normal login by session shape alone. The app's root navigator gates access to `(app)` purely on session presence, so without further care a recovery session would satisfy that guard and drop the user straight into the main app before they've actually changed their password.

**Decision:** Track password recovery as its own `AuthStatus` value (`passwordRecovery`), set only in response to Supabase's authoritative `PASSWORD_RECOVERY` auth event ŌĆö never inferred from session presence alone, and never treated as equivalent to `authenticated`. `reset-password` is gated behind `Stack.Protected guard={status === 'passwordRecovery'}`, structurally separate from the `authenticated` guard on `(app)`, so a recovery session can reach only the reset-password screen and nothing else in the app. After a successful `updateUser({ password })`, the recovery session is explicitly ended (`signOut()`) and the user is redirected to `/login` (with a success flag) rather than into `(app)` ŌĆö password reset always ends in an explicit, intentional sign-in with the new credential, not an implicit continuation of the recovery session.

**Consequences:** The auth state machine stays clean and closed: `passwordRecovery` is only ever entered via the real Supabase event and only ever exited via sign-out, so it can't be spoofed by a malformed deep link or silently upgraded to a full session. This does mean password reset costs the user one extra sign-in step compared to auto-login after reset ŌĆö accepted as the safer default. The platform-specific parts of the flow (deep-link URL construction, reading `code`/`type` from the callback route) are isolated in `src/lib/supabase/auth.ts` and `auth-callback.tsx`; the state machine itself, the Supabase call shapes (`resetPasswordForEmail`, `updateUser`), and the redirect-to-login behavior are platform-agnostic and intended to carry over to the future web implementation with only those isolated parts needing a web-specific rewrite (browser redirect URL instead of a deep link, `detectSessionInUrl`-based session capture instead of manual code exchange).

## StyleSheet + theme tokens

**Context:** Need a consistent visual language (colors, spacing) across components without committing early to a heavier styling library.

**Decision:** Keep React Native's built-in `StyleSheet` API, backed by the existing token system in `src/constants/theme.ts` (`Colors`, `Spacing`, `Fonts`) and the `useTheme` hook. `theme.ts` already documents NativeWind, Tamagui, and Unistyles as known alternatives.

**Consequences:** Styling stays dependency-free and close to React Native primitives. This is deliberately deferred, not rejected outright ŌĆö revisit only if the current approach causes real friction as the component set grows (see `docs/roadmap.md`).

## Program Creation Wizard: shared store instead of local state, deferred write

**Context:** Web's creation wizard collects all 5 steps (name, type, level, days count, then a per-day exercise picker) in the parent component's local React state and writes to Supabase immediately after the final step ŌĆö three sequential, non-transactional inserts (`programs` ŌåÆ `program_days` ŌåÆ `program_exercises`) with a real, confirmed orphan-row risk on partial failure, and (separately) an edit flow that's actually broken in web today: `updateTrainingProgram` exists but is never called, so editing an existing program silently creates a duplicate rather than updating it. Mobile's wizard is being built incrementally across sprints, and Sprint 19's exercise picker is expected to be a separate route (reusing the exercise wiki's browsing UI), not a step embedded in the same screen ŌĆö a pushed route has no clean way to hand data back into a specific still-mounted parent screen's local state; route params only flow parent ŌåÆ child.

**Decision:** Wizard data (name, type, level, days count ŌĆö exercises to be added when Sprint 19 needs them) lives in a temporary Zustand store (`src/stores/program-wizard-store.ts`), not local component state ŌĆö exactly the "in-progress form drafts" use case this project's own Zustand decision already anticipated, not a new pattern. The current step number stays local `useState` in the wizard screen, since it's pure UI/navigation state, not data that needs to survive a screen swap. The store resets on three triggers only ŌĆö starting a fresh Create flow, an explicit Cancel, and (in the later sprint that adds it) a successful final Create ŌĆö deliberately _not_ on navigating between wizard steps or returning from the future exercise picker, which Expo Router's `Stack` supports naturally since it keeps the wizard screen mounted underneath any screen pushed on top of it. Sprint 18 itself performs **no Supabase writes at all** ŌĆö steps 1ŌĆō4 only collect data locally; the single write (`programs` + `program_days`, and later `program_exercises`) is deferred until exercise selection is also complete, specifically to avoid ever creating a partially-empty program row, which is a stronger guarantee than web provides today.

**Consequences:** Any future multi-screen flow with the same "a later screen needs to hand data back to an earlier one" shape should default to a small, scoped Zustand store rather than route params or React Context ŌĆö Context would be a second, redundant pattern for a problem this app already has an established solution for. The final write landed in Sprint 20 (see "Program Creation Wizard: final write" below), keeping the same sequential-insert approach web uses rather than introducing a transaction, but with a narrower blast radius than web's (nothing is written until the entire flow, including exercises, is ready) and with app-layer cleanup of already-committed rows on partial failure, which web itself doesn't have. Building true edit/update semantics later still means writing that logic from scratch ŌĆö web's own `updateTrainingProgram`, while more complete than what's wired up, is unused/undertested and has a real data-shape bug (it expects `program_exercises.id` values but the wizard state actually holds raw `exercise_id`s), so it should be treated as reference material at most, not a source to port directly.

## Exercise Picker: pushed route, shared browsing logic, scoped Confirm/Cancel

**Context:** Web's exercise picker is a per-day modal (full-screen CSS overlay) with its own muscle-group tabs and, critically, no search box at all ŌĆö mobile's Explore screen already has both search and muscle-group filtering built (Sprints 12ŌĆō16), and the picker needs the exact same browsing behavior, just with tap-to-select instead of tap-to-navigate. Building the picker meant either duplicating Explore's fetch/filter logic and search/filter UI, or extracting it ŌĆö with a second real consumer now needing the identical logic, this crossed from "one screen's implementation detail" into "actual duplication," the same bar this project has applied consistently (`formatProgramType`/`isNotFoundError` were extracted the same way, once a second consumer appeared, not before).

**Decision:** `src/hooks/use-exercise-browser.ts` now holds the fetch/filter logic that used to live directly in `explore/index.tsx` (unchanged in shape ŌĆö same `loadState`, same client-side muscle-group + search filtering) ŌĆö Explore was rewritten to consume it rather than the picker copying Explore's code. `src/components/exercise-search-bar.tsx` and `src/components/muscle-group-filter.tsx` are the same extraction for the search input row and the filter-pill row, both now shared between Explore and the picker. The picker itself (`src/app/(app)/programs/exercise-picker.tsx`) is a pushed `Stack.Screen` registered as a new sibling in the existing `programs/_layout.tsx` ŌĆö no modal presentation, matching this app's zero existing precedent for it and avoiding the accidental-dismiss risk a swipeable modal would introduce for a screen whose whole point is a deliberate Confirm-vs-Cancel choice. Selection state is local `useState` in the picker, seeded once from `program-wizard-store.ts` on mount and written back via `setDayExercises` only on Confirm ŌĆö Cancel calls nothing but `router.back()`, touching the store not at all, mirroring web's own Cancel/Confirm boundary exactly (the one piece of web's design kept as-is). The wizard's step 4 now renders one card per day (`Array.from({ length: daysCount })`, reading `days[dayIndex]` from a single whole-array store subscription rather than one hook call per day inside the loop, which would violate the Rules of Hooks), each with its own "Add/Edit exercises" action passing that day's real index as a route param ŌĆö replacing the earlier hardcoded `dayIndex=0` placeholder from the first implementation phase.

**Consequences:** Any future screen needing the exercise catalog with search/filtering should consume `useExerciseBrowser` rather than re-fetching ŌĆö there are now two real call sites proving the extraction was warranted, not a guess. The picker's selection UI (numbered badges via array position, `theme.backgroundSelected` for the selected-card background) intentionally reuses this app's own established "selected" visual language rather than porting web's opacity-based approach, and shows a running "Confirm (N)" count web itself lacks ŌĆö a small, deliberate improvement, not an oversight. The final wizard-submit button (`handleFinishPress` at the time) was still a placeholder as of this sprint ŌĆö this sprint only made the full local-state flow (name ŌåÆ type ŌåÆ level ŌåÆ days ŌåÆ per-day exercises) work end to end; the actual write was wired up in Sprint 20 (see "Program Creation Wizard: final write" below), which also renamed the handler to `handleCreatePress`.

## Program Creation Wizard: final write ŌĆö sequential inserts kept, with best-effort cleanup

**Context:** Sprint 18 deferred the final write and flagged the sequential-insert, no-transaction risk inherited from web as accepted-but-not-solved, noting no Postgres RPC/transaction existed anywhere in this project. Landing the actual write meant deciding explicitly, for the first time in this project, between two strategies: match web's three sequential inserts (`programs` ŌåÆ `program_days` ŌåÆ `program_exercises`) or introduce this project's first Postgres RPC/transaction to make the write atomic. Web's own insert order isn't just a data dependency (each step needs the previous step's generated ids) ŌĆö it's required by the live RLS INSERT policies on `program_days`/`program_exercises`, both of which authorize via an `EXISTS` subquery through the parent row(s), confirmed directly against the Supabase project rather than assumed.

**Decision:** Sequential client-side inserts were kept (`createProgram` in `src/lib/supabase/programs.ts`), not moved to an RPC. Introducing this project's first SQL migration/RPC tooling ŌĆö no `supabase/migrations` directory or authored SQL exists anywhere in this repo ŌĆö to solve one feature's atomicity was judged disproportionate, consistent with this project's repeated pattern of deferring new architectural surface until a real, not speculative, need exists (the same standard applied to TanStack Query every time it's come up). The orphan-row risk this inherits from web is mitigated, not eliminated, at the application layer: on a `program_days`/`program_exercises` failure, `cleanupOrphanedProgram` deletes whatever was already committed before re-throwing the original error, rather than leaving a partially-created program behind. This mitigation is simpler than it first appeared ŌĆö live inspection of the foreign-key constraints (`program_days.program_id ŌåÆ programs.id` and `program_exercises.day_id ŌåÆ program_days.id`, both confirmed `ON DELETE CASCADE` via a direct `pg_constraint` query against the linked project, not inferred) confirmed that cascade delete makes explicit `program_exercises` cleanup unnecessary: deleting `program_days` alone cascades through any `program_exercises` rows underneath it, so cleanup only needs two explicit deletes (`program_days`, then `programs`), not three. Web's own `deleteProgramWithRelations` deletes all three tables explicitly, which is safe but redundant, not a sign that cascade delete is absent.

The wizard integration (`src/app/(app)/programs/create.tsx`) also settled three UX questions: the Create button (and Cancel, and the step-4 in-page Back link) are disabled while the write is in flight, using the same `isSubmitting`-gated pattern already used by `reset-password.tsx`/`forgot-password.tsx`; success resets the wizard store and then navigates via `router.replace` (not `push`) to the new Program Detail screen ŌĆö an improvement over web, which always redirects to a list ŌĆö with `replace` chosen specifically so back-navigation from the new program doesn't land on a stale, reset wizard; failure leaves all wizard data untouched and re-enables the same Create button as the retry path, with the error rendered via this app's existing `ŌØī {message}` + `themeColor="danger"` pattern already used in `programs/[id].tsx` and `exercise-picker.tsx`.

**Consequences:** This decision should be revisited, not silently carried forward, once Edit or Delete implementation begins ŌĆö both have messier failure modes than Create (Edit's positional diff can leave old and new data commingled on a partial failure; a future `delete_program` RPC would remove the reverse-order manual delete entirely) and may tip the balance toward an RPC for reasons specific to those features, not just because the sequential pattern is already in place here. Any future multi-table write in this project should default to the same pattern established here ŌĆö sequential inserts plus app-layer compensating cleanup on failure ŌĆö until a genuine transactional need, not just tidiness, justifies introducing this project's first RPC/migrations workflow.

## Program Deletion: single cascading delete, ownership gate, local platform branch

**Context:** Web's `deleteProgramWithRelations` manually deletes `training_history` ŌåÆ `program_exercises` ŌåÆ `program_days` ŌåÆ `programs`, in that order, mirroring the reverse of its own create function's insert order. Sprint 20 had already confirmed `program_days.program_id ŌåÆ programs.id`, `program_exercises.day_id ŌåÆ program_days.id`, and `training_history.day_id ŌåÆ program_days.id` are all `ON DELETE CASCADE`. A fresh check for Sprint 21 (`pg_policies`/`pg_class.relrowsecurity` against the linked project) additionally found `program_days`/`program_exercises` have RLS enabled with only `SELECT`/`INSERT` policies ŌĆö no `UPDATE` or `DELETE` policy exists on either table. That means web's own explicit `program_exercises`/`program_days` deletes are themselves silent RLS no-ops today (0 rows affected, no error); its function only actually works end-to-end because Postgres executes `ON DELETE CASCADE` actions as the table owner, which bypasses RLS (confirmed `relforcerowsecurity = false` on both tables) ŌĆö the final `programs` delete's cascade is doing all the real work regardless of what came before it. Program Detail (Sprint 17) also had no ownership check at all, deliberately deferred until an owner-only affordance actually existed.

**Decision:** `deleteProgram(id)` in `src/lib/supabase/programs.ts` is a single `supabase.from('programs').delete().eq('id', id)` call ŌĆö not a port of web's four-step version, since three of those four steps are either redundant (cascade already covers them) or non-functional under the live RLS policies as they stand today. `ProgramDetailRow`/`PROGRAM_DETAIL_QUERY` gained `user_id` (nullable, matching the live column) specifically to support an ownership check: `programs/[id].tsx` renders the Delete action only when `program.user_id === user?.id` (`useAuthStore`), the same comparison web's `isMyProgram` makes. Confirmation uses an inline `Platform.OS === 'web'` branch directly in `programs/[id].tsx` ŌĆö `Alert.alert` on native, `window.confirm` on web (verified necessary by reading `react-native-web`'s actual installed source: `Alert.alert()` is a literal no-op there, not assumed). This stays inline rather than becoming a shared `src/lib/` module: no second real consumer exists today ŌĆö Edit isn't a destructive action, and a Global Programs "remove" confirm is speculative and out of current roadmap scope ŌĆö matching this project's standing rule to extract only once a second real consumer exists, not because a platform split "might" be reused later. On success, `router.replace('/programs')` ŌĆö not `push` or `back()` ŌĆö verified against the exact v57 Expo Router docs that `replace` removes the current screen from navigation history entirely, so back-navigation cannot return to the just-deleted Program Detail screen; `replace` was chosen over `back()` specifically because `back()`'s correctness would depend on assuming this screen was always reached via a single push from the list, an assumption `replace`'s explicit destination doesn't need.

**Consequences:** Any future destructive-confirm need, if and when one actually appears, should default to the same inline `Platform.OS` branch unless a second real consumer makes extraction concretely worthwhile ŌĆö this is now the explicit precedent for that call, not just an implicit norm. The RLS gap found here (`program_days`/`program_exercises` missing `UPDATE`/`DELETE` policies) is a hard blocker for Edit, not just a Delete concern ŌĆö Edit cannot port web's `updateTrainingProgram` as-is, since its `.update()`/`.upsert()` calls on `program_exercises` would be silent no-ops under the same policies today; this needs its own explicit decision (new RLS policies vs. some other approach) before Edit is implemented. Program Detail's ownership check (`user_id === user?.id`) is now established and should be reused as-is when Edit adds its own owner-only affordance, rather than reimplemented.

## Google Sign-In and Register parity: browser-redirect OAuth, functional-only scope

**Context:** Web already has working Google Sign-In (`supabase.auth.signInWithOAuth`) and collects `firstName`/`lastName` at registration, populating `full_name`/`avatar_url` user metadata; mobile had neither. A full audit compared every auth screen against web in detail (colors, buttons, inputs, spacing) and found mobile's visual language is a different, self-consistent system already used across every other screen (Programs, Profile, Explore) ŌĆö not a variant of web's. Bringing auth screens to full visual parity would mean either diverging mobile's own auth screens from the rest of the app, or updating `theme.ts`/shared button and input styling app-wide, both far larger than an authentication sprint.

**Decision:** Scope was deliberately narrowed to functional parity only ŌĆö Google Sign-In and the register metadata gap ŌĆö with `theme.ts`, `AuthTextField`, shared button styling, typography, and colors explicitly untouched. For Google Sign-In, the browser-redirect strategy was chosen over the native `@react-native-google-signin/google-signin` SDK (Supabase's generic top recommendation): `signInWithGoogle` (`src/lib/supabase/auth.ts`) uses `expo-web-browser` (already a dependency since the project's first commit, previously unused, so no new native rebuild was needed) via `WebBrowser.openAuthSessionAsync`, reusing the existing `authCallbackUrl()`/`exchangeCodeForSession` PKCE plumbing built for password reset and email confirmation ŌĆö consistent with this project's standing pattern of avoiding new native dependencies/config until a simpler path is proven insufficient. It does **not** route through `/auth-callback` on native ŌĆö `openAuthSessionAsync` captures the redirect directly in the calling code, unlike the email-triggered flows that rely on a genuine OS-dispatched deep link. Re-checking Supabase's current docs before implementing (rather than relying on the initial research) surfaced one addition the original plan missed: on the web target, the same `/auth-callback` route needs a web-only, module-scope `WebBrowser.maybeCompleteAuthSession()` call so a Google sign-in opened as a popup can signal completion back to its opener ŌĆö confirmed unsupported (not just inert) on native, so it's guarded by `Platform.OS === 'web'`. The Google icon reuses web's exact official SVG asset (`assets/images/google-icon.svg`) rather than a rasterized fallback ŌĆö confirmed both that Metro's default `assetExts` already includes `svg` and that `expo-image` (already used throughout the app) renders SVG natively on iOS/Android/web, so no new dependency (`react-native-svg`) was needed. Register's `firstName`/`lastName` fields and their validation mirror web's actual `firstNameOptions`/`lastNameOptions` (`lib/validations.tsx`), and `signUpWithPassword` now builds `full_name`/`avatar_url` metadata identically to web's `registerUserWithEmail` ŌĆö including reusing the same third-party `ui-avatars.com` generator web relies on, since web has no shared helper for this either, just an inline expression mobile now mirrors rather than abstracts.

Manual verification on web surfaced a real double-exchange race not caught by initial testing: `openAuthSessionAsync` has no OS-level session capture on web (unlike `ASWebAuthenticationSession`/Custom Tabs on native), so it opens a real popup that genuinely navigates to and loads the actual `/auth-callback` route ŌĆö the same screen password reset and email confirmation use, which already unconditionally called `exchangeCodeForSession` on its own. That raced against `signInWithGoogle()`'s own exchange in the opener window; whichever attempt was still in flight when the opener's `dismissPopup()` closed the popup failed with a raw `TypeError: Failed to fetch` (the request's execution context torn down mid-flight, not a clean HTTP rejection from Supabase). Fixed by capturing `maybeCompleteAuthSession()`'s return value (previously called and discarded) and skipping the popup's own exchange whenever it reports `'success'` ŌĆö that result is only ever `'success'` when this load genuinely is the OAuth popup, in which case the opener is already about to perform the one real exchange. This has no effect on native (the value is always `null` there, guarded by the same `Platform.OS === 'web'` check) or on password reset/email confirmation on web (`maybeCompleteAuthSession` reports `'failed'` for a direct page load, so their existing exchange behavior is unchanged).

**Consequences:** Manually verified working end-to-end on the custom development build and web, including the double-exchange fix above ŌĆö this is this project's first OAuth flow of any kind, and the browser-redirect pattern (not the native SDK) is now the established precedent for any future provider, unless a concrete need for native-SDK-quality UX arises later. Two prerequisites were required outside this repo and confirmed necessary but not independently checkable by any tooling available during implementation: the mobile redirect URL present in Supabase's dashboard redirect-URL allow-list, and Supabase's fixed callback URL registered in Google Cloud Console ŌĆö both are external configuration this repo cannot track or verify, only rely on. Any future screen that both (a) opens a popup-based web flow via `openAuthSessionAsync` and (b) redirects back to a route with its own independent side effects should check for this same class of race, not assume a single call site. Full visual/design-system parity with web (colors, button/input styling, typography) remains explicitly deferred, tracked in `docs/roadmap.md` as its own future cross-app redesign, not folded into this sprint.

**Context:** Web's persistent `Header`/`AppShell` wraps every authenticated page; mobile has never had an equivalent ŌĆö each screen manages its own top-of-screen back link/title ad hoc, duplicated near-verbatim across most of them (the `SafeAreaView` + centered themed container shell repeats across every auth screen; the back-link `Link`/`ThemedText` pair repeats across every detail screen). Expo Router offers two genuinely different ways to introduce shared chrome: a Stack-level `screenOptions.header` (automatic for every screen under that Stack) or a plain component each screen renders in its own body. This project has already deliberately chosen `headerShown: false` everywhere, specifically "for visual consistency with the rest of the app" (see the Exercise detail screen entry above) ŌĆö reversing that would mean re-litigating a decision already made, not just adding a component.

**Decision:** Two separate, independently-adoptable components ŌĆö `ScreenLayout` (`src/components/screen-layout.tsx`, the shared shell) and `ScreenHeader` (`src/components/screen-header.tsx`, a back/title/right three-slot row) ŌĆö kept as plain screen-rendered components, not Stack header options. The title slot uses `smallBold`, matching Exercise Picker's pre-existing compact header-title size, not the app's 48px hero `title` style ŌĆö screens with a real page-hero title (Program Detail's program name) keep rendering it separately, below the header. Both components were adopted immediately rather than introduced unused: `ScreenHeader` in `programs/[id].tsx`, `explore/[id].tsx`, and `programs/exercise-picker.tsx`; `ScreenLayout` in `confirm-email.tsx` (its container style is an exact match, copied directly from it). Separately, `app-tabs.web.tsx`'s tab bar chrome still showed "Expo Starter" branding and a `docs.expo.dev` link ŌĆö leftover, never-touched Expo Router starter-template boilerplate (only the tab `href`s had ever been updated to this app's real routes). Replaced with the app's actual name; no logo image was introduced, since web's own logo asset (`public/logos/logo.png`) turned out to be a large glow/hero graphic, not sized for inline header use, and mobile's own app icon is still Expo's unmodified default ŌĆö real logo/icon artwork is a separate, unstarted concern, not a decision made here.

**Consequences:** Any future screen needing a back/title header row should default to `ScreenHeader` rather than hand-rolling another `Link`+`ThemedText` pair; any future screen with the plain centered-shell shape should default to `ScreenLayout`. Adopting `ScreenHeader` changed Exercise Picker's title from right-aligned (in a two-child `space-between` row) to horizontally centered (`ScreenHeader`'s title slot is `flex: 1, textAlign: 'center'`) ŌĆö a small, deliberate visual change, not an oversight. Neither component needed a `.web.tsx` split ŌĆö both are pure composition of already-cross-platform primitives, no platform-specific behavior involved. Real branded logo/icon artwork remains an open, unstarted gap, tracked in `docs/roadmap.md`.

## Home tab: global training-history feed, ported as-is

**Context:** Web's Home page is literally one component ŌĆö a global, cross-user activity feed of every completed workout entry, unfiltered by current user, no pagination, no personalization. Mobile's Home tab was still 100% untouched Expo Router starter-template boilerplate. Sprint 25's own research phase recommended building a personalized programs summary in that slot instead (reusing `getPrograms`/`ProgramCard`, already built), given the feed shows _other_ users' data and mobile has no way to contribute to it yet (Workout Logging doesn't exist). That recommendation was explicitly overridden: the direction given was to port the feed web actually has, not substitute a different concept, so this entry documents what was built, not the alternative that was proposed and declined.

**Decision:** `src/lib/supabase/training-history.ts` (new) holds `getTrainingHistoryFeed()`, joining `training_history` ŌåÆ `profiles` via the single live-confirmed FK (`fk_user`) ŌĆö no ownership filter, matching web's `fetchAllTrainingHistories` exactly. `getExerciseIdsForProgramExercises` was added to `programs.ts` (co-located with the `program_exercises` table it queries, matching this project's one-file-per-table convention) to resolve `training_history.values`' keys (`program_exercise_id`s) down to real exercise names via the already-existing `getExercises()`/`localizeExercise`, mirroring web's two-step `fetchProgramExerciseMap` + `fetchExercisesByIds`.

Two correctness improvements over web's own code, found by checking the live schema rather than assuming it matches web's implicit assumptions: `training_history.user_id` is nullable (0 of 14 current rows are, but the column allows it), so `profiles` is modeled as nullable in the Zod schema and the UI falls back to "Unknown" ŌĆö web's own code has no such guard and would break on a row like that. Sorting was moved server-side: chained `.order('date', {ascending:false}).order('created_at', {ascending:false})` rather than porting web's client-side `.sort()` comparator ŌĆö confirmed via the installed `@supabase/postgrest-js` source (each `.order()` call appends to the request's order parameter rather than overwriting it) that this produces the exact same `ORDER BY date DESC, created_at DESC` semantics as web's two-level comparator, not assumed equivalent.

Two deliberate, smaller adaptations: dates use `toLocaleDateString(undefined, ...)` (device locale), matching the convention `profile.tsx` already established, rather than porting web's hardcoded `'uk-UA'`, which isn't part of this app's actual i18n system and wouldn't match anything else already built. The feed's avatar/username renders as plain, non-interactive info rather than web's link to `/profile/{userId}` ŌĆö mobile has no route to view another user's profile at all, and building one is a distinct feature beyond "port the feed," not folded in here.

One implementation gotcha worth recording for future similar code: an initial version mixed a non-promise value into `Promise.all([value, promiseA, promiseB])`, which caused TypeScript to lose positional tuple inference entirely and union all three result types together ŌĆö not a runtime bug, but a confusing type error. Restructured as a plain sequential `.then()` chain instead, which avoided the issue entirely; future multi-fetch loading code in this app should default to that shape rather than mixing already-resolved values into `Promise.all`.

**Consequences:** The feed could only grow via web at the time this landed ŌĆö mobile had no logging feature yet to post new entries into it, making this read-only parity, not full parity, until Workout Logging landed (see "Workout Logging: dedicated orchestration module, ref-based draft autosave" below). The avatar-with-fallback-initial UI now exists in two places (`profile.tsx` and this feed) with no shared component between them; not extracted here since that would mean touching `profile.tsx`, outside this sprint's stated scope ŌĆö a small, real duplication a future sprint could address once (if) a third consumer makes the case clearer, matching this project's standing extraction bar. The `/profile/[id]` gap is now tracked explicitly in `docs/roadmap.md` rather than left implicit.

## Workout Logging: dedicated orchestration module, ref-based draft autosave

**Context:** Web's `completeDay` (`lib/trainingData.ts`) writes a workout across three tables ŌĆö `exercise_logs` (one row per exercise), `training_history` (one denormalized JSON row per day), then deletes that day's `exercise_drafts` ŌĆö fed by autosave-on-blur drafting. Mobile had only the read half built (`getTrainingHistoryFeed`, for Home) and no write path on any of the three tables. The requirement for this sprint was explicit: `exercise_drafts`/`exercise_logs`/`training_history` stay independent, single-table modules that never import each other, and one new orchestration module owns the cross-table sequencing ŌĆö a stronger constraint than this project's existing "one file per table" convention alone would have implied on its own.

**Decision:** `src/lib/supabase/exercise-drafts.ts` (`getDrafts`, `saveDraft`, `deleteDrafts`) and `src/lib/supabase/exercise-logs.ts` (`insertExerciseLogs`) are new, table-scoped, matching the existing convention exactly; `training-history.ts` gained a sibling write (`createTrainingHistoryEntry`) alongside its existing read. A new `src/lib/supabase/workout.ts` is the _only_ place that imports across all three ŌĆö `completeDay(userId, dayId, date, values)` sequences insert-logs ŌåÆ insert-history ŌåÆ delete-drafts, awaited in order (not `Promise.all`), matching web's own step order exactly, including writing the same filtered (non-empty-value) map to both `exercise_logs` and `training_history`. Unlike web's `completeDay`, which swallows errors and returns a boolean, mobile's throws ŌĆö matching how every other Supabase function in this codebase already behaves, not a literal port of web's error-handling shape.

The logging UI (`src/components/workout-log-form.tsx`, mirroring web's `TrainingProcessing`) surfaced two real, non-obvious bugs during review, both fixed before landing rather than merely noted:

1. Its draft-load effect originally depended on the caller-supplied `exercises` array by reference. `programs/[id].tsx` builds that array inline in JSX, so it's a new reference on every render of that screen for any reason ŌĆö an unrelated re-render (e.g. the Delete button's status changing) would refire the draft fetch and silently overwrite an in-progress, unsaved keystroke. Fixed by deriving a stable string key (`programExerciseIds.join(',')`) as the effect dependency instead of the array itself.
2. The initial "read the value to save on blur" implementation read from `values` React state, which isn't guaranteed to reflect the very last keystroke by the time blur fires (`onChangeText`'s state update and the blur event can land in the same batch). The fix first attempted was reading `event.nativeEvent.text` directly off the blur event ŌĆö checked against the installed React Native types before landing it, which showed `TextInput.onBlur` is typed as `BlurEvent` (`TargetedEvent`-based) in this version, carrying no text payload at all, so that approach doesn't compile. Landed instead as a `valuesRef` ref, mutated synchronously alongside every `setValues` call ŌĆö refs update immediately regardless of React's render/commit timing, so `handleBlur`/`handleComplete` read from `valuesRef.current` rather than the `values` state closure.

Home's feed (`(app)/index.tsx`) was also changed ŌĆö `fetchData` now runs via `useFocusEffect` (`useCallback`-wrapped, matching the pattern React Navigation's own docs recommend to avoid resubscribing every render) instead of a mount-only `useEffect`, so a freshly completed workout appears on return to Home without requiring a full screen remount. `fetchData` still never resets to `'loading'` itself, so a refocus refetch swaps data in silently rather than flashing the loading state over already-visible content.

Per-day workout history display (read-only past entries, refresh-after-completion) was deliberately scoped out of this sprint to keep it a reviewable, independently-shippable slice ŌĆö the read half already exists for Home's global feed, but a day-scoped `training_history` read and its UI are still open, tracked in `docs/roadmap.md`.

**Consequences:** Any future multi-table write workflow in this project should default to the same shape established here ŌĆö independent table modules plus one dedicated, otherwise-empty-of-Supabase-calls orchestration module ŌĆö rather than folding sequencing logic into any one table's file. Any future autosave-on-blur (or similarly timing-sensitive) input pattern should default to the ref-mirrors-state approach here rather than reading directly from React state or assuming a specific native event carries the current value ŌĆö both were tried and rejected for this exact feature. `workout.ts` currently has exactly one consumer (`WorkoutLogForm`); when history display lands, its day-scoped read belongs in `training-history.ts` alongside the feed read and the new write, not in `workout.ts`, which stays write-orchestration-only by design (see "Per-Day Workout History: parallel-loadable query, pure presentational component" below ŌĆö landed exactly this way).

## Per-Day Workout History: parallel-loadable query, pure presentational component

**Context:** Workout Logging's own entry above deliberately scoped out history display to keep that sprint reviewable ŌĆö the write half (`exercise_logs`/`training_history`) existed, but `programs/[id].tsx` had no way to show past entries per day, and completing a workout had nowhere to visibly land beyond the global Home feed. `training_history` has no direct `program_id` column, only `day_id` (FK to `program_days`), which raised a real design question: a naive `getTrainingHistoryForDays(dayIds)` would require knowing a program's day ids first, forcing a waterfall (fetch program ŌåÆ extract day ids ŌåÆ fetch history) rather than joining the same `Promise.all` `programs/[id].tsx` already uses for `getProgramDetail`/`getExercises`.

**Decision:** `getTrainingHistoryForProgram(programId)` (new, in `training-history.ts`, not `workout.ts` ŌĆö the standing "workout.ts stays write-orchestration-only" decision held) filters through the embedded `program_days!inner(program_id)` relation instead, needing only the `programId` already available from route params ŌĆö this is what lets it join the same `Promise.all` as the other two fetches with zero sequential dependency, confirmed by the query's own shape rather than assumed. Results are grouped into `Record<dayId, TrainingHistoryEntry[]>` before returning, matching the same map-returning convention already established by `getDrafts`/`getExerciseIdsForProgramExercises` ŌĆö the sole consumer only ever needs "this day's entries," so grouping happens once in the data layer rather than being re-implemented in the screen. `history` was added to `LoadState`'s existing `success` variant in `programs/[id].tsx`, not a separate `useState`: same lifecycle as `program`/`exerciseNames` (loaded together, meaningful together), and embedding it makes an inconsistent render (fresh program paired with stale history from a previous navigation) impossible to represent, not just unlikely.

`WorkoutHistory` (new, `src/components/workout-history.tsx`) is intentionally pure presentation ŌĆö it receives already-loaded `entries` and an `exercises` name list as props and fetches nothing itself, reusing the exact `WorkoutLogExercise` shape `WorkoutLogForm` already defined for the same per-day exercise list (imported via `import type`, not duplicated). It renders as a vertical list of past entries, most recent first (already sorted server-side, same two-column `.order()` pattern as `getTrainingHistoryFeed`) ŌĆö a deliberate departure from web's `TrainingHistory`, which renders dates as columns and exercises as rows in a grid that only works at desktop width; the same information is rebalanced into a list rather than pixel-ported, consistent with this project's standing precedent for adapting layout to platform rather than porting it literally.

Refresh-after-completion: `WorkoutLogForm` gained an optional `onComplete?: () => void`, called once inside its existing `handleComplete` success branch (not a new effect, not a new dependency anywhere) ŌĆö `programs/[id].tsx` passes a `refreshHistory` function that re-fetches the whole program's history (matching web's own `loadAllHistory`, which also refetches everything rather than just the completed day) and merges it into `loadState` via a functional update. Because `onComplete` is only ever called imperatively and never placed in an effect's dependency array, it can't reintroduce the class of refetch-loop bug `WorkoutLogForm`'s own draft-load effect already had to be fixed for in the previous sprint ŌĆö and that earlier fix (the `programExerciseIdsKey`-derived dependency) is what makes the history refresh's incidental re-render of `programs/[id].tsx` safe: it rebuilds the `exercises` array passed to `WorkoutLogForm` again, but the derived-key dependency means this doesn't refire the draft fetch.

**Consequences:** Any future data-layer function facing this exact shape ŌĆö needing to filter one table via a relationship to a parent id, without wanting to fetch the parent first ŌĆö should default to checking whether PostgREST's embedded-relation filtering (`relation!inner(column)` + `.eq('relation.column', value)`) can remove that dependency, as it did here, rather than assuming a sequential fetch is required. `training-history.ts` now holds one read for the global feed, one read for a single program's per-day breakdown, and one write ŌĆö all three call sites distinct, with no cross-imports needed between them. This closes out the Workout Logging feature's read loop entirely; the only remaining tracked gaps in this area are Program Edit (still RLS-blocked) and Global Programs/Complexes (unscoped), both in `docs/roadmap.md`.

## Global Programs (Complexes): push-navigated browser, reused write path, no order column

**Context:** Web's Complexes feature (`lib/complexesData.ts`, `app/training/complexes/`) lets any user browse trainer-authored "global" programs and copy one into their own `programs` via `addGlobalProgramToUser`/`removeGlobalProgramFromUser`, rendered as an MUI `Accordion` list with an inline Add/Remove toggle per row; trainer-only authoring of new global programs was explicitly out of scope for this sprint. An architecture review compared two mobile UX shapes ŌĆö porting the Accordion versus a push-navigated list ŌåÆ detail screen ŌĆö and two data-layer questions: whether `getUserGlobalProgramMap` could be folded into a single combined query with the catalog fetch, and whether `global_program_exercises` has any deterministic ordering column beyond the never-verified assumption that web's lack of an `.order()` call was itself meaningful.

Live schema verification (the same isolated-workdir `supabase db query --linked` convention as every prior sprint) confirmed: `global_programs` has RLS enabled with a public `SELECT` policy; `global_program_days`/`global_program_exercises` have RLS disabled entirely (no auth needed to browse, matching `exercises`/`muscle_groups`); `programs.source_global_program_id` **does** carry a real FK to `global_programs.id` (`ON DELETE NO ACTION`); and ŌĆö checked via a full `information_schema.columns` query, not inferred from web's code ŌĆö `global_program_exercises` has exactly three columns (`id`, `day_id`, `exercise_id`), no `created_at`, no sequence/position column of any kind, unlike `program_exercises`, which has `order_index` for exactly this purpose.

**Decision:** The Accordion was rejected in favor of a push-navigated list (`src/app/(app)/programs/complexes/index.tsx`) ŌåÆ detail screen (`[id].tsx`), folder-first with its own nested `_layout.tsx` (matching `explore/`'s precedent, chosen up front since this sprint needed both screens from the start, unlike `programs/[id].tsx`, which needed no further nesting). This is a native-mobile-appropriate adaptation, not a literal port ŌĆö mobile has zero accordion/expand-in-place precedent anywhere in this app, every existing browse-then-view flow already uses push navigation, and an Accordion's variable-height nested lists inside a `FlatList` would fight React Native's virtualization in ways web's simple layout-shifting `<div>` never has to contend with.

`src/lib/supabase/complexes.ts` (new) holds `getGlobalPrograms()` (flat catalog fetch, mirroring `getPrograms`) and `getGlobalProgramDetail(id)` (nested `global_program_days ŌåÆ global_program_exercises` query, mirroring `getProgramDetail`'s tree-select shape). Exercises within a day are deliberately **not** sorted ŌĆö there is no column to sort by, and sorting by `id` (a random `gen_random_uuid()`, not a sequence) would fabricate the appearance of an intended order the data doesn't actually encode, which is worse than transparently matching web's own identical lack of an `.order()` call here. Only day order (`day_number`) is sorted, same as `getProgramDetail`.

`addGlobalProgramToUser`/`removeGlobalProgramFromUser` do no raw table writes of their own ŌĆö they delegate entirely to the already-proven `createProgram`/`deleteProgram` in `programs.ts`, rather than porting web's manual per-table insert/delete steps. This required one small addition: `CreateProgramInput` gained an optional `sourceGlobalProgramId`, written straight into the same initial `programs` insert as `source_global_program_id` ŌĆö the FK confirmed above ŌĆö so a copy is linked to its source in the same atomic step `createProgram` already performs, with no separate follow-up update. `removeGlobalProgramFromUser` reuses `deleteProgram`'s single cascading delete for the same reason `deleteProgram` itself doesn't port web's three-table version: `program_days`/`program_exercises` have no DELETE RLS policy, so explicit steps there would be silent no-ops (see "Program Deletion" above).

`getUserGlobalProgramMap(userId)` stays a separate flat query against `programs` (`user_id` + non-null `source_global_program_id`), not folded into `getGlobalPrograms()` via the confirmed FK ŌĆö an explicit, temporary choice, not a rejection of the idea: the architecture review concluded a combined embedded-relation query is architecturally sound now that the FK is verified, but doing so wasn't part of the approved Sprint 28 scope. The function is commented accordingly so a future combined-query refactor is a small, well-understood change, not a rediscovery. The map serves two consumers ŌĆö the list screen's "Added" badge (via a new optional `badge?: ReactNode` slot on `ProgramCard`) and the detail screen's Add/Remove button state and "View in My Programs ŌåÆ" link ŌĆö fetched once per screen via the same independent-`Promise.all` shape already used elsewhere (`programs/[id].tsx`, Home's feed).

**Consequences:** Any future "does the user own a copy of X" check should default to this same flat-query shape unless a combined query is explicitly chosen and re-verified against the FK at implementation time ŌĆö this entry is the reference for that decision, not a settled precedent to copy blindly. If `global_program_exercises` ever needs real, editable ordering, that requires a schema change (`order_index` added, populated at trainer-authoring time, mirroring `program_exercises`), not a client-side ordering heuristic ŌĆö trainer authoring itself remains unscoped, tracked in `docs/roadmap.md` alongside Program Edit.

**Context:** The Home feed (Sprint 25) has shown a non-interactive avatar/username on every entry since it was built, specifically because there was nowhere to navigate to ŌĆö tracked explicitly in `docs/roadmap.md` as a known gap, not an oversight. Web's own `/profile/[id]` (`app/profile/[id]/page.tsx`) renders a profile header plus a Friends section (`Friends.server.tsx`); the Friends/social area was explicitly deferred back in Sprint 23 and stays deferred here ŌĆö only the header and a "this user's programs" section were in scope. Research found both were unusually close to already built: `getProfileById` (built in the very first custom-table sprint) was already documented as not special-casing "own profile" fetching, and `getPrograms(userId)` (built for the Programs tab) already accepted an arbitrary `userId`, mirrored exactly by web's own `fetchUserPrograms(id)` gating only its Create-button/empty-copy on `isMyProfile`, not the fetch itself.

**Decision:** `profile.tsx` was migrated to `profile/_layout.tsx` (new nested Stack, `headerShown: false`, mirroring `explore/_layout.tsx`) + `profile/index.tsx` (the existing "my profile" screen, content unchanged) + `profile/[id].tsx` (new). Git recorded this as an 81%-similarity rename, not a delete+add, reflecting that it's a relocation with edits rather than a new screen. No changes were needed to `app-tabs.tsx`/`app-tabs.web.tsx` ŌĆö both reference the tab by the route name `"profile"`, which Expo Router resolves to a folder's `_layout.tsx` transparently, exactly as it already does for `programs`/`explore`.

Two components were extracted, both crossing the threshold this project has consistently used elsewhere ŌĆö a second or third _real_ consumer, not a preemptive abstraction:

- `Avatar` (`src/components/avatar.tsx`) consolidates the image-or-initial-fallback rendering that was independently duplicated in `profile.tsx` and the Home feed (each with its own local `initialFrom` helper); `profile/[id].tsx` became the third consumer, the one that made the duplication worth resolving. It owns only the circle ŌĆö sizing, image-vs-fallback, and the initial ŌĆö never layout direction or press handling, which stay screen-specific (the Home feed's horizontal row vs. the profile screens' vertical header).
- `ProgramsList` (`src/components/programs-list.tsx`) extracts just the `FlatList`-of-`ProgramCard`s block, previously only in `programs/index.tsx`, now shared with `profile/[id].tsx`. A higher-level fetch-owning `ProgramsSection`/`ProgramsContent` was explicitly considered and rejected during architecture review: Favorites/Shared Programs/Trainer Programs (raised as hypothetical future consumers) would each need a different query ŌĆö Trainer Programs most likely maps to `GlobalProgramRow`/`global_programs`, not `ProgramRow`/`programs`, a different type entirely ŌĆö so a fetch-owning component would either become a speculative variant-dispatcher or need rewriting per feature anyway. It would also have introduced a second, independent loading/error cycle nested inside a screen that already resolves everything into one combined `LoadState`, which no screen in this app does today. `ProgramsList` therefore stays fetch-free and state-free; `programs/index.tsx` was refactored to consume it too, so there's one implementation, not a new one alongside the old inline block.

**No data-layer changes were made** ŌĆö `getProfileById` and `getPrograms` are reused exactly as they already existed, a first for this project; every previous sprint touched at least one data-layer file.

**Ownership** is `loadState.profile.id === user?.id` ŌĆö a synchronous comparison against `useAuthStore`, no extra fetch, mirroring web's own `currentId === profile.id` check. It gates the Sign Out button and the Create-Program button/empty-state wording (own: "You don't have any programs yet."; another user's: "This user hasn't created any programs yet."); the profile header and the programs list itself render identically either way, with no redirect when `id` happens to equal your own ŌĆö again matching web, which conditionally shows a Settings link rather than redirecting.

**Privacy:** `loadState.profile.email` is intentionally never rendered on `[id].tsx`, even though `getProfileById` returns it and RLS permits reading any profile's email (per `profile.ts`'s own decision log). This is a deliberate choice made during implementation, beyond what the approved architecture plan explicitly specified, called out with an inline comment at the render site rather than left silently absent ŌĆö own-profile's email display on `profile/index.tsx` is unaffected (that's the current user's own email, sourced from the auth session, not another user's row).

**Home feed navigation:** the feed's avatar/username block (`(app)/index.tsx`) is now a `Pressable`, disabled when `item.profiles` is null (the row's profile relation is nullable per Sprint 25's schema), pushing to `/profile/${item.profiles.id}`.

The v57 Native Tabs reference was also checked for cross-tab push safety (navigating from a screen under the Profile tab to `/programs/[id]`, registered under the Programs tab) and confirmed: the docs state JS-driven navigation "still navigates to the tab" even when that tab's native trigger is disabled ŌĆö stronger evidence than the ordinary enabled case, and this app's first exercise of cross-tab imperative navigation.

**Consequences:** Any future screen reachable from more than one place should default to `onBackPress`/`canGoBack()`-guarded `back()`, not a fixed `backHref` ŌĆö `ScreenHeader`'s two back mechanisms now have a clear rule for which to use, not just two options. `Avatar`/`ProgramsList` are now the default for any future avatar or program-list rendering respectively; a fetch-owning `ProgramsSection` remains explicitly not built and shouldn't be introduced speculatively ŌĆö only once a second real, built feature needs the exact same fetch-and-render shape, not merely a hypothetical one. Tapping a program from someone else's profile reuses `programs/[id].tsx` unmodified, since it already hides Delete/`WorkoutLogForm` behind its existing ownership check ŌĆö confirming that screen was already safe for this use case before this sprint touched it.

## App Shell Consolidation: one ScreenLayout, no new abstractions

**Context:** A research pass across the whole app (not just the auth screens `ScreenLayout` was originally built for) found the same shell duplicated near-verbatim in three shapes: a centered, form-like container (the four auth screens, byte-identical to `ScreenLayout`'s existing implementation); a top-aligned container ending in a `FlatList` (`explore/index`, Home, `programs/index`, `programs/complexes/index`, `programs/exercise-picker`); and a `ScrollView`-based container for single-column detail content (`explore/[id]`, `programs/[id]`, `programs/complexes/[id]` ŌĆö these three were **100% identical**, not just similar). `ScreenLayout` already existed for exactly this purpose but wasn't consistently adopted, leaving most screens hand-rolling the same `SafeAreaView` + `ThemedView`/`ScrollView` pair it was meant to replace. The audit also surfaced two genuine bugs, not just duplication: `profile/index.tsx` used `Spacing.six` for its top padding and gap where every other screen uses `Spacing.four`, and `profile/[id].tsx` combined `BottomTabInset + Spacing.four` at the container level while also rendering `ProgramsList`, whose own `FlatList` already adds a trailing `Spacing.four` ŌĆö a real, if small, double bottom-padding.

**Decision:** Rather than introducing a second or third layout component for the list-bearing and scrollable shapes, `ScreenLayout` was extended with three narrow, additive props, all defaulting to its original behavior so no existing consumer changed without an explicit new prop:

- `scroll?: boolean` (default `false`) ŌĆö swaps the internal `ThemedView` for a `ScrollView` (with `theme.background` applied internally, mirroring what the three detail screens previously did by hand), covering the third shape.
- `justify?: 'center' | 'flex-start'` (default `'center'`) ŌĆö `'center'` preserves the auth screens' original centered behavior; `'flex-start'` covers every top-aligned screen (lists, the wizard, both profile screens).
- `contentStyle?: StyleProp<ViewStyle>` ŌĆö an escape hatch for the per-screen values this shell has never standardized (`paddingTop`, `paddingBottom`, and `gap` where a screen genuinely differs from the four-unit default), merged after the base styles, the same role `Button`'s own `style` prop already plays. Each screen's `contentStyle` reproduces only its own actual prior values ŌĆö nothing was normalized or rounded to a "nicer" shared number where screens didn't already agree (e.g. `auth-callback.tsx` kept its pre-existing `gap: Spacing.three` rather than being pulled to `Spacing.four`).

Future additions to `ScreenLayout` should require at least two real consumers before expanding the public API.

A fourth prop, `align?: 'stretch' | 'center'`, was deliberately **not added** ŌĆö `auth-callback.tsx` was the only screen needing `alignItems: 'center'`, so it takes that through `contentStyle` instead of widening `ScreenLayout`'s permanent API for a single call site. This follows the same principle the whole sprint was scoped around: consolidate existing duplication into the one component that already existed, but don't grow that component's surface area beyond what's actually earned by more than one real consumer ŌĆö the same bar this project has applied to every other extraction (`Avatar`, `ProgramsList`, `isNotFoundError`, etc.).

One implementation detail caught during self-review, not during planning: the three original `scroll`-shape screens never had `flex: 1` on their `ScrollView`'s `contentContainerStyle` ŌĆö only the non-scroll shape's container did. `ScreenLayout`'s shared base style needed splitting (`container` without `flex`, plus a separate `fill` style applied only in non-scroll mode) so adopting `scroll` wouldn't silently introduce a layout property those three screens never had.

Two intentional UI fixes rode along with the adoption, both already called out in the research/plan and approved before implementation: `profile/index.tsx`'s padding/gap normalized to `Spacing.four`, and `profile/[id].tsx`'s bottom padding reduced to `BottomTabInset` alone (removing the redundant `+ Spacing.four` now that `ProgramsList` supplies its own trailing space) ŌĆö both real, if minor, visual changes, not accidental side effects of the refactor.

**Consequences:** Any future screen shell need should default to `ScreenLayout` with the appropriate `scroll`/`justify`/`contentStyle` combination rather than a new hand-rolled `SafeAreaView` pair ŌĆö this sprint is the reference for which combination fits which shape. If a fourth genuinely distinct shape or a second real consumer for `alignItems: 'center'` appears later, extending the prop set then is reasonable; adding it speculatively now was not. `programs/create.tsx`'s wizard Cancel now goes through `ScreenHeader` (`onBackPress`, no title) rather than an inline link, matching `exercise-picker.tsx`'s existing Cancel ŌĆö this closes the last screen in the app that didn't use `ScreenHeader` for its back affordance.

## Program editing, metadata-only: reused wizard, no structural changes, RLS gap still open

**Context:** A full architecture investigation (research doc, approved before implementation) found web's own edit feature non-functional in four distinct ways ŌĆö its save handler never actually calls its update function (silently creates a duplicate program instead), the dormant update function conflates `exercise_id` with `program_exercises.id` (masked today only because a second bug's fallback happens to compensate), it never deletes days removed by shrinking the day count, and its exercise diffing matches by array position rather than row identity ŌĆö which would silently reassign historical `training_history`/`exercise_logs` entries to the wrong exercise if it were ever actually exercised. None of this was ported, even in spirit. Separately, live-reconfirmed RLS state (unchanged since Sprint 21) still has no `UPDATE`/`DELETE` policy on `program_days`/`program_exercises`, and every cascade from those tables down through `exercise_drafts`/`exercise_logs`/`training_history` is destructive ŌĆö so any structural edit (add/remove/reorder days or exercises) needs both a deliberate RLS decision and identity-preserving (not positional) diffing, neither of which this sprint attempts.

**Decision:** Scope was deliberately narrowed to metadata only ŌĆö `title`/`type`/`level` via a new `updateProgramMetadata(id, userId, updates)` in `programs.ts`, a single `.update()` against `programs` (RLS disabled on that table, confirmed live) that never touches `program_days`/`program_exercises`. The `.eq('user_id', userId)` filter on that update is defense-in-depth, not an enforced constraint ŌĆö `programs` has no RLS, so this mirrors the same trust model `deleteProgram` already relies on for this exact table, made explicit here rather than a new pattern.

Rather than a parallel edit screen, `programs/create.tsx` was extended with an optional `programId` param (passed from a new ownership-gated "Edit program" action on `programs/[id].tsx`, gated identically to Delete). The existing `useProgramWizardStore` is reused as-is ŌĆö no new store ŌĆö since `name`/`type`/`level` are identical in shape to what Create's steps 1ŌĆō3 already manage; `days`/`daysCount` are simply never read or written in edit mode. Leak-prevention between modes rests on one guarantee: the screen's mount effect always calls `resetWizard()` first, unconditionally, before any conditional prefill runs ŌĆö so whichever mode last populated the store, the next mount (edit or create) always starts from a fully blank slate before `name`/`type`/`level` are set again. Edit mode's prefill (via the existing `getProgramDetail`) only ever calls `setName`/`setType`/`setLevel` ŌĆö `days`/`daysCount` stay at `reset()`'s defaults (`[]`/`null`) for the entire session, so there's no path for a prior Create draft's days to survive into an edit, or vice versa. Reset triggers are unchanged from Create's existing three (fresh mount, explicit Cancel, post-success) ŌĆö no new trigger was added for edit mode.

The wizard's flow caps at step 3 in edit mode: step 4 (days count + exercise picker) is unreachable, and step 3's action becomes "Save changes" (calling `updateProgramMetadata`) instead of advancing further. On success, `router.replace('/programs/${id}')`, mirroring Create's own redirect-to-detail pattern.

One implementation-time correction: the first draft called `setPrefillState` synchronously inside the mount effect, which this project's `react-hooks/set-state-in-effect` convention (state only set inside `.then()`/`.catch()`) correctly rejects. Fixed by relying on `useState`'s initializer for the synchronous "start in loading" case (mirroring `programs/[id].tsx`'s own `id ? {state:'loading'} : ...` pattern) and moving the retry button's synchronous reset into its own event handler, where synchronous `setState` is the established norm.

**Consequences:** Structural editing (add/remove/reorder days or exercises) remains explicitly deferred, blocked on the same RLS gap identified in Sprint 21 and reconfirmed this sprint ŌĆö that decision (new policies vs. an alternative) still needs to be made deliberately, not assumed, before any future sprint attempts it. Any future edit-mode prefill into this wizard should follow the same reset-first-then-prefill sequencing established here, and any future structural diffing must match by row identity (`program_exercises.id`), never array position ŌĆö the exact web bug this sprint's research documented as the reason not to.

## Structural program editing: identity-based diff, INSERT+DELETE only, no reorder

**Context:** Sprint 31 deferred structural editing (adding/removing days or exercises) entirely, blocked on the RLS gap first identified in Sprint 21. This sprint resolved it, but not by simply adding back everything web's own edit feature implies ŌĆö web's actual `updateTrainingProgram` (still dead code, never called by `handleSave`) has four confirmed bugs on top of never running at all: it conflates `exercise_id` with `program_exercises.id` (masked only by a second bug's fallback), never deletes days removed by shrinking the day count, and diffs exercises by array position rather than row identity ŌĆö which would silently reassign historical `training_history`/`exercise_logs` entries to the wrong exercise if it ever actually ran. None of that logic was ported.

Before implementing, an explicit question was raised and verified rather than assumed: does structural editing actually need an `UPDATE` policy on `program_days`/`program_exercises`, or only `INSERT`/`DELETE`? Answer, confirmed by re-reading the app's own UI: **`program_days` never needs UPDATE** ŌĆö the day-count control only ever grows/shrinks at the tail (no reorder, no arbitrary-position removal), so an existing day's `day_number` never changes. **`program_exercises` only needs UPDATE if reordering an already-selected exercise must be non-destructive** ŌĆö since delete+reinsert cascades away that exercise's history before any "replacement" row could exist, there is no way to change an existing slot's position without either an UPDATE or losing its history. Given the exercise-picker has no drag-reorder UI today (the only way to visually reorder is deselect-then-reselect), the explicit decision was made to **not support exercise/day reordering this sprint**, which means every write is either a pure insert or a pure delete ŌĆö no `UPDATE` policy needed on either table at all.

**Decision:** The wizard store's `days: string[][]` became `WizardDay[]` (`{ id: string | null; exercises: { id: string | null; exerciseId: string }[] }[]`) ŌĆö `id: null` meaning "not yet persisted." Create mode is simply the case where every id stays null; `createProgram` itself is unchanged, with the wizard mapping the richer shape down to plain exercise-id arrays at its one call site. `exercise-picker.tsx`'s Confirm handler reconciles the final selection against the store's _current_ slots for that day by `exerciseId` (never position), so an exercise that stays selected keeps its real row id regardless of how many times it was toggled within the session or where it ends up in the list ŌĆö this is what makes the diff safe by construction rather than by convention.

`updateProgramStructure` (replacing Sprint 31's `updateProgramMetadata`, which is now removed ŌĆö superseded, not left alongside) re-fetches `getProgramDetail` fresh at save time and diffs against it: days beyond the new day count are deleted (cascade covers their exercises/drafts/history ŌĆö no separate cleanup needed); within kept days, any existing exercise whose id no longer appears is deleted; new days and new exercise slots are inserted, with new exercises always appended after existing ones in a day (`order_index` = current max + 1, ...) since kept rows' `order_index` is never rewritten. One disclosed consequence of the no-reorder decision: a newly-added exercise always lands after all kept exercises in that day, even if the picker showed it interleaved among them ŌĆö true interleaving would require renumbering kept rows, which the no-UPDATE decision rules out.

A destructive-removal confirmation (reusing the `Platform.OS`-branched `Alert`/`window.confirm` pattern from Program Deletion) fires only when the diff actually finds something to remove, via a `confirmRemoval` callback the data layer calls before writing anything ŌĆö keeping `programs.ts` free of UI concerns while still doing only one fetch. Returning `false` aborts with zero writes performed (not even the `programs` row update); the call site treats this as a return to idle, not a success or an error.

**Web alignment, checked directly rather than assumed:** web's actual `SelectExercisesStep`/`CreateEditProgram` were read directly, not inferred from the earlier bug research alone. Its day-count-only model (no reorder, no arbitrary removal) already matched what was planned. Its `isValidProgram` check (every day needs Ōēź1 exercise before the final button enables) was a real rule this app hadn't adopted ŌĆö now applied to both Create and Edit. Its wizard always enters at step 1 even for edit (no skip-ahead) ŌĆö adopted here too, reversing an earlier plan to jump straight to step 4, since that friction wasn't actually mobile-specific (web has the identical linear flow). Two things found on web were deliberately **not** ported: it has no removal confirmation anywhere (a direct consequence of its edit path never actually deleting anything, not a considered UX decision ŌĆö porting "no confirmation" would have shipped mobile's real, working delete with no warning), and its final-step button always reads "Create" even in edit mode (inconsistent, unrelabeled ŌĆö an artifact of the feature being unfinished, not a decision to match).

**A process note worth recording:** the first implementation of `updateProgramStructure` issued `.delete()` calls against `program_days`/`program_exercises` without first re-verifying that a `DELETE` policy actually existed on either table. It didn't ŌĆö only `SELECT`/`INSERT` were live, meaning the deletes would have silently no-op'd in production (Supabase doesn't error on a zero-row delete) while the UI reported success. This was caught before committing, not after. The fix was the two policies below, applied live, verified narrower than (not equal to, and never broader than) the existing `INSERT` policies ŌĆö they reuse only the personal-program-ownership branch (`programs.user_id = auth.uid()`), omitting the trainer/`global_programs` branch INSERT has, since global program editing remains unbuilt and out of scope:

```sql
create policy "Allow delete program_days" on program_days
  for delete using (exists (select 1 from programs p where p.id = program_days.program_id and p.user_id = auth.uid()));

create policy "Allow delete program_exercises" on program_exercises
  for delete using (exists (select 1 from program_days d join programs p on p.id = d.program_id where d.id = program_exercises.day_id and p.user_id = auth.uid()));
```

**Consequences:** Any future write path touching `program_days`/`program_exercises` should verify its required RLS policies live before writing code that depends on them, not after ŌĆö the same "verify, don't assume" discipline this project applies to schema shape should extend explicitly to policy existence for new operations, which this sprint's process gap shows isn't automatic just because the discipline exists elsewhere. Exercise/day reordering remains unsupported; if it's ever wanted, it needs its own decision (an `UPDATE` policy on `program_exercises` at minimum) rather than being assumed available. `updateProgramMetadata` is gone ŌĆö any future metadata-only need should extend `updateProgramStructure`'s shape rather than reintroducing a narrower parallel function.

## Settings: non-blocking theme preference, first real caller for `updateProfileById`

**Context:** `updateProfileById`/`ProfileUpdate` were built in this project's very first custom-table sprint but had never had a single caller ŌĆö no screen let a user change their name or theme. Web's own `/settings` (name editing, split into first/last name fields) and its header `Menu` (a separate, instant-apply dark/light toggle persisted to `profiles.dark`) were the two pieces ported; language selection was explicitly excluded ŌĆö there's no i18n infrastructure anywhere in this app, and a language picker with no translations behind it would be dead UI, not a feature.

**Decision ŌĆö theme preference storage and fetch timing:** `themePreference: boolean | null` was added to `useAuthStore` (not a separate store ŌĆö it's current-user data, exactly what that store already exists to own) rather than gating app startup on fetching it. The first proposal was to fetch it inside `initialize()` and block `RootNavigator`'s render on it, reusing the exact mechanism session restore already uses. That was deliberately reconsidered and rejected during architecture review: unlike session restore, this fetch is sequentially dependent on already knowing the user id, making it a genuine _additional_ network round trip on every cold start, not a one-time cost ŌĆö paid by every user, including the majority who've never set an explicit preference and would see zero visual difference either way. The adopted design instead fetches in the background, non-blocking: a `refreshThemePreference(userId)` helper (owned entirely by `auth-store.ts`, not `AuthProvider`, which stays a deliberately thin one-shot mounting shim) fires fire-and-forget from both places a session can newly become known ŌĆö the initial `getCurrentSession()` resolution and the ongoing `subscribeToAuthChanges` callback (a fresh sign-in mid-session needs the same fetch, not just cold-start restore). A stale-response guard (`useAuthStore.getState().user?.id === userId`) discards a response that arrives after the user has signed out or a different user has since signed in. `themePreference` resets to `null` on sign-out, so a second user on a shared device never briefly inherits the previous one's theme.

**Failure behavior, by construction:** because nothing awaits this fetch, there is no path by which a failure (offline, timeout, server error) can leave the app stuck ŌĆö `.catch()` swallows the error silently and `themePreference` simply stays whatever it already was (`null` on first load), the same OS-scheme fallback a user with no preference gets. No retry loop exists; Settings' own `getProfileById` call (needed anyway to seed the name form) doubles as a natural, low-cost resync point if the background fetch failed or hadn't resolved yet.

**Decision ŌĆö theme toggle is not optimistic UI, by explicit choice, not oversight:** `handleThemeSelect` calls `updateProfileById` first and only calls `setThemePreference` (the only line that changes what actually renders) inside `.then()`, after the write succeeds. During the request, the only visible change is the pills becoming briefly disabled ŌĆö colors and pill-selection stay exactly as they were until the server confirms. On failure, `themePreference` was never touched, so there is nothing to roll back; an error line appears and the pills re-enable for the user to retry manually. This was verified against the actual code (not just the intended design) and explicitly confirmed as the preferred behavior ŌĆö optimistic UI (flipping immediately, reverting on failure) was considered and declined for this screen.

**Decision ŌĆö shared scheme resolution:** a new `resolveEffectiveScheme(osScheme, preference)` pure function in `use-theme.ts` is now the one source of truth for "what theme is actually in effect" ŌĆö an explicit preference always wins; the OS scheme is only the fallback when no preference has ever been set. Both `useTheme()` and `_layout.tsx`'s `ThemeProvider` selection (React Navigation's own `DarkTheme`/`DefaultTheme`) now route through it; `_layout.tsx` also switched from calling `react-native`'s raw `useColorScheme` to this app's own custom hook (matching `useTheme()`'s existing source), fixing a small pre-existing inconsistency between the two that had never mattered before (no preference could exist yet) but genuinely could have caused disagreement once one did.

**Decision ŌĆö name-splitting bug fix, not a straight port:** web seeds its name form via `profile.username?.split(' ')` destructured into exactly `[firstName, lastName]`, which silently drops every word beyond the second for a three-plus-word name ŌĆö a confirmed bug (how JS array destructuring behaves), not a design choice. Mobile uses `firstName = words[0]`, `lastName = words.slice(1).join(' ')` instead, preserving every word while still recombining identically for the common two-word case ŌĆö the same "don't port a confirmed bug just because web has it" standard already applied in Program Editing's research. The dirty-check gating Save uses React Hook Form's own `isDirty` rather than a hand-rolled comparison (web's own approach) ŌĆö a small improvement RHF provides for free, not extra complexity for its own sake.

**Decision ŌĆö `nameSchema` extraction:** the same first/last-name validation rule (min 3 characters, letters only) existed only inline in `register-form.tsx` until this sprint. Settings became a genuine second consumer, crossing this project's established extraction bar ŌĆö it's now `src/lib/validation.ts`, imported by both forms instead of duplicated.

**Consequences:** Any future need for "the current user's own data available app-wide, cheaply" should default to extending `useAuthStore` the way `themePreference` did, not introducing a new store, unless the data genuinely doesn't belong to the authenticated-user lifecycle. Any future background fetch on the same pattern should default to non-blocking-with-silent-failure unless a concrete, specific reason requires blocking startup ŌĆö the reasoning that reversed the original blocking proposal here is the reference case. `resolveEffectiveScheme` is now the only place scheme precedence should ever be computed; a third call site should never reimplement it inline.

## Maintenance 01: removed two unrestricted `friends` RLS policies

**Context:** Social Area research (ahead of Sprint 34) found two policies on `public.friends` scoped to role `public` with `qual: true`: `"Allow realtime delete"` (DELETE) and `"Allow realtime select"` (SELECT) ŌĆö both named as if required for Supabase Realtime. Neither was: Realtime authorizes row broadcast per-subscriber using a table's SELECT policy alone, never its DELETE policy, and every real client subscribes with an authenticated session, already covered by the separate `"Allow selecting friends for logged-in user"` policy (`qual: true`, `authenticated`). The realtime-scoped SELECT policy was therefore a redundant duplicate of an already-open policy, and the realtime-scoped DELETE policy was strictly harmful ŌĆö since Postgres OR's together all permissive policies for a command, its unconditional `qual: true` silently overrode the two properly ownership-scoped DELETE policies sitting next to it, letting any client delete any friendship row.

**Decision:** Dropped both policies; added nothing. Re-verified live afterward: DELETE now has exactly the two ownership-scoped policies (`Allow deleting accepted friendships`, `Allow deleting own pending friend requests`); SELECT now has exactly `Allow selecting friends for logged-in user`; INSERT/UPDATE unchanged; `pg_publication_tables` still lists `friends` under `supabase_realtime`, confirming Realtime was unaffected.

**Consequences:** No application code changed ŌĆö this was a live database-only fix, standalone from Sprint 34. The pre-existing, separately-flagged fully-open SELECT policy (any authenticated user can read any row) and the either-party UPDATE policy (sender or recipient can change `status`) remain untouched and open, out of this task's scope.

## Maintenance 02: added the missing DELETE policy for declining incoming friend requests

**Context:** Sprint 35 research (comparing web's `declineFriendRequest` in `lib/friendData.ts` against live RLS) found that `friends` had exactly two DELETE policies ŌĆö one for a sender deleting their own pending request (Cancel), one for either party deleting an `accepted` friendship (Remove) ŌĆö and neither covers the _recipient_ deleting an incoming `pending` row (Decline). Since Supabase doesn't error on a zero-row RLS-filtered delete, web's Decline button has been silently no-oping in production: the UI removes the item locally and reports success, but the row remains in the database forever. This is a live production bug on web today, not a hypothetical mobile risk.

**Decision:** Added `"Allow declining incoming pending friend requests"` ŌĆö `FOR DELETE TO authenticated USING (friend_id = auth.uid() AND status = 'pending')` ŌĆö mirroring the existing sender-side policy's shape exactly, scoped to the recipient side instead. Deliberately narrow: it only ever matches a row where the caller is the _recipient_ of a _still-pending_ request, so it cannot be used to delete an accepted friendship or a request the caller merely sent (those remain covered by the other two policies). Re-verified live after applying: all three DELETE policies present, correctly scoped `TO authenticated` (an initial application accidentally defaulted to `PUBLIC` by omitting the `TO authenticated` clause ŌĆö caught and corrected before this report, not left as a residual gap), and `friends` still present in the `supabase_realtime` publication, confirming no effect on Realtime.

**Consequences:** No application code changed. Mobile's forthcoming Sprint 35 `declineFriendRequest` can now rely on this policy actually deleting the row rather than silently failing ŌĆö a prerequisite this project's data layer should not have to route around defensively; the query should simply trust that a successful DELETE call means the row is actually gone, since RLS now correctly matches this case.

## Sprint 34, Users & Friends foundation: nested under Profile, no Requests, no Realtime yet

**Context:** The Social Area (Users/Friends/Friend Requests) was the last major web feature area with no mobile equivalent. A dedicated research-and-architecture phase (not summarized here in full) compared web's implementation directly rather than assuming it, live-verified the `friends` schema/RLS, and confirmed `friends` is the only table in this project's `supabase_realtime` publication. Three decisions were made deliberately before implementation:

- **`users.ts` was rejected in favor of extending `profile.ts`.** The initial research proposal split by feature (Users/Friends/Requests) rather than by table ŌĆö inconsistent with this codebase's actual convention, where `exercises.ts` already keeps `getExercises()` (list-all) and `getExerciseById()` (single row) in one file because they hit the same table. Browsing all users is `profiles` with no `.eq('id', ...)` filter, not a different table ŌĆö so `getAllProfiles()` was added to `profile.ts`; only `friends.ts` is a new file, since `friends` is a genuinely distinct table.
- **Realtime was deferred**, not because "simpler first" is automatically better, but because the Friend Requests screen ŌĆö the only feature realtime actually drives ŌĆö is itself deferred to a later sprint. Building it now would mean designing this app's first-ever subscription lifecycle (mount/unmount, `AppState` foreground/background, sign-out teardown) with no working reference to adapt (web's own subscription has no cleanup at either call site ŌĆö a confirmed bug, not a pattern to port) in the same sprint as an RLS change touching the same table. When Requests is built, it should sit behind a `{ requests, status, refresh() }`-shaped hook so adding realtime later means patching that hook's internals, not changing the consuming screen.
- **Social was nested under the existing Profile tab, not a fifth `NativeTab`.** Consistent with every prior addition (Settings, Complexes, program editing) ŌĆö none grew the tab bar. Promoting to a full tab later, if Social grows, is cheap and additive; removing a tab later is the harder, more visible change to walk back.

A standalone maintenance task (documented above) removed two unrestricted `friends` RLS policies before this sprint began, independent of its schedule.

**Decision:** `src/lib/supabase/friends.ts` exports `Friendship`/`getFriendshipsForUser` (both directions in one query), `sendFriendRequest`, and a single `deleteFriendship(friendshipId)` used for **both** "Cancel request" and "Remove friend" ŌĆö the two ownership-scoped DELETE policies already differentiate by the row's actual status/parties server-side, so the client-side call is identical either way; a second, identically-bodied function would only be duplication. `getFriendshipState(friendships, viewerId, otherUserId)` is a pure helper (alongside the data functions, mirroring `formatProgramType`/`formatProgramLevel` living in `programs.ts`) deriving `'none' | 'outgoing' | 'incoming' | 'accepted'` from the two-directional list. `'incoming'` (someone else's pending request to you) has no action yet ŌĆö Requests/accept is a later sprint ŌĆö so it renders as a disabled "Pending" label on the Users screen rather than a misleading button.

Two new screens, `profile/users.tsx` and `profile/friends.tsx`, both added to the existing `profile/_layout.tsx` Stack and linked from `profile/index.tsx` alongside the existing Settings link. A new shared `UserCard` (`src/components/user-card.tsx`) ŌĆö avatar, name, tap-through to `/profile/[id]`, optional right-side action slot mirroring `ProgramCard`'s existing `badge` slot ŌĆö is used by both screens rather than building two near-identical rows; it is not built on a shared base "Card" component, since none exists for the two existing card-like components (`ProgramCard`, `UserCard`) to extract into, matching this project's standing preference against premature abstraction. Both screens use `FlatList` (via direct use here, not a `ProgramsList`-style wrapper, since the two screens' rows differ by whether an action slot is passed) rather than a plain mapped list, since `ScreenLayout`'s non-scroll container needs a `FlatList` child to remain scrollable for a long list ŌĆö the same reason `ProgramsList` itself is a `FlatList`.

Actions on the Users screen are non-optimistic (button disables during the request, refetches `friendships` only ŌĆö not the whole profiles table ŌĆö on success), matching this project's established Settings-era default rather than web's optimistic add/cancel. "Remove friend" alone gets a destructive-action confirmation (the same `Platform.OS`-branched `Alert`/`window.confirm` pattern as Program Deletion) since it's the only action here that undoes an existing relationship; "Add friend"/"Cancel request" do not, since neither destroys data web itself doesn't also let you redo trivially.

**Consequences:** Friend Requests (incoming list, accept/decline) and Realtime remain fully unbuilt ŌĆö the next sprint should build Requests behind the hook-boundary described above, with Realtime as a clean, later addition to that same hook rather than a parallel new sprint concern layered on top. `getAllProfiles`/`getFriendshipsForUser` (whole-table fetches) are the natural reuse point for that sprint rather than new queries.

**Functional testing:** verified on the actual custom development client (`com.corefit.mobile`, rebuilt this same session after finding the previously-installed dev client was stale ŌĆö see the Android dev-client-vs-Expo-Go troubleshooting note below), not Expo Go. Confirmed working: Users list (ordering, exclusion of the viewer's own row, avatar/initials rendering), Add Friend and Cancel Request (both cross-checked directly against the live `friends` table, not just the UI), row-tap navigation to `/profile/[id]`, back navigation, loading states, and error+retry on both screens (forced via disabling network mid-session). Two scenarios were intentionally left untested rather than seeded with throwaway production data: **Remove Friend** (requires an existing `accepted` friendship, which can only be created via Accept ŌĆö the Friend Requests sprint's own feature) and the **incoming "Pending" disabled-label state** (requires a second real account sending a request to the test account). Both are recorded here as integration scenarios to verify once the Friend Requests sprint makes them reachable through the UI, rather than being faked via direct data manipulation now.

**Android dev-client note:** during this sprint's testing, the emulator had a stale custom dev client installed (last built long before this session, predating the current `expo`/`react-native` versions), which crashed on launch with `ReferenceError: Property 'MessageQueue' doesn't exist` ŌĆö a native/JS runtime mismatch, not an app bug.
