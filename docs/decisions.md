# Architecture Decisions

Lightweight decision log for the choices that shape CoreFit Mobile's foundation. Each entry is Context (the problem), Decision (what we chose), Consequences (what that implies going forward).

## Expo Router

**Context:** Need a routing solution for a universal (iOS/Android/web) Expo app that scales past a handful of screens without hand-wired navigation config.

**Decision:** File-based routing via `expo-router`, with routes under `src/app` and `experiments.typedRoutes` enabled in `app.config.ts`.

**Consequences:** New screens are added by creating files, not by registering them in a navigator config. Typed routes catch invalid `href`s at compile time. Route groups and layouts (`_layout.tsx`) are the mechanism for shared chrome (tabs, headers).

## Supabase

**Context:** Need auth, a relational database, and file storage without standing up and operating custom backend infrastructure.

**Decision:** Supabase as the backend — Postgres database, built-in auth, storage, and row-level security (RLS) for authorization. Considered Firebase briefly; Supabase was chosen for SQL/Postgres (easier relational modeling for workout/progress data) and a first-class TypeScript client.

**Consequences:** Client code talks to Supabase directly from the app via its JS client (no custom API server layer to start). Authorization logic lives in Postgres RLS policies, not application code. `EXPO_PUBLIC_SUPABASE_URL` and `EXPO_PUBLIC_SUPABASE_ANON_KEY` (see `.env.example`) are the only credentials the client needs — the anon key is safe to embed in the bundle by design.

## TanStack Query

**Context:** Screens will need to fetch, cache, and revalidate data from Supabase without each screen re-implementing loading/error/refetch state.

**Decision:** `@tanstack/react-query` as the server-state layer, wrapping Supabase calls in query/mutation hooks.

**Consequences:** Server data is cached and deduplicated across screens automatically. Loading/error UI follows one consistent pattern. Server state (from Supabase) and client state (UI-only) stay clearly separated — server state does not belong in Zustand.

## Profile data layer (Foundation)

**Context:** The app needs a typed, validated way to read and update the `profiles` table before any profile UI is built. The `profiles` schema isn't checked into either this repo or the web repo as migrations/DDL — the only authoritative source is the live Supabase project. `profiles` is also the first custom (non-auth) table the mobile app talks to, so this is the first real decision point for how table access should be structured, ahead of TanStack Query landing.

**Decision:** `src/lib/supabase/profile.ts` is the sole place that touches `supabase.from('profiles')`, mirroring `src/lib/supabase/auth.ts`'s existing convention exactly — no separate repository/service split, one file, plain Promise-based functions (`getProfileById`, `updateProfileById`). The `profiles` schema is defined as a Zod object matching the live database (columns, types, and nullability confirmed directly against the Supabase project via `supabase db query`, not inferred from either app's client code), and every Supabase response is parsed through it before being returned. `ProfileUpdate` is scoped to only the columns that are actually meant to be user-editable (`username`, `avatar_url`, `language`, `dark`) — `email` and `is_trainer` are excluded even though the table's RLS `UPDATE` policy doesn't itself restrict them, because neither is meant to be changed this way (`email` belongs to the auth flow; `is_trainer` is backend/admin-managed).

No `useProfile` hook or any client-side cache/store is introduced yet — this layer is deliberately Promise-only. Hooks and caching are deferred until TanStack Query is introduced (see above), once there are enough server-state features to justify it; adding a throwaway loading hook now would just be removed and replaced later.

**Consequences:** Any future table beyond `profiles` should default to this same single-file convention (schema + types + thin functions, colocated) unless it grows enough distinct concerns to justify splitting. When TanStack Query lands, `profile.ts`'s functions become the query/mutation functions passed into hooks — nothing about this layer's shape needs to change, only a hook layer gets added on top. Because RLS on `profiles` allows public `SELECT` access (including unauthenticated reads, and including `email`), callers of `getProfileById` must not assume the requester is viewing their own data — this file intentionally does not special-case "own profile" fetching.

## Exercise data layer and client-side filtering

**Context:** The web app's exercise "wiki" (`exercises` + `muscle_groups` tables, confirmed live: 61 exercises, 7 muscle groups, RLS disabled on both) issues a fresh Supabase query every time the user taps a muscle-group filter tab, plus a separate round-trip to resolve the tapped tab's hardcoded English name to a `muscle_group_id` before it can even run that query. With only 61 rows total, the entire table is cheap to hold in memory at once, and mobile's first exercise screen (`src/app/(app)/explore.tsx`) needed a filtering approach to decide on before this could be built at all.

**Decision:** `src/lib/supabase/exercises.ts` fetches the full `exercises` and `muscle_groups` tables once, unfiltered (`getExercises()` takes no parameters), mirroring `profile.ts`'s single-file convention and colocating the raw Zod schema/types with a `localizeExercise` mapping function in the same file rather than a separate module. Muscle-group filtering (`selectedMuscleGroup: string | null`, where `null` means "All") and the eventual name/description text search are both applied client-side, in-memory, in the screen itself — not as Supabase query parameters, and "All" is never written to or read from the database as a real `muscle_groups` row, it's purely a UI-level `null`. `ExerciseRow` (the raw per-language DB shape) and `LocalizedExercise` (the UI-facing, single-language-resolved shape produced by `localizeExercise`) are kept as two distinct types on purpose, even though only the `'en'` locale is used anywhere today — this is the one seam where real i18n would plug in later without changing the data layer or the fetch functions.

**Consequences:** Selecting a filter tab is instant and offline-safe (no network dependency once the initial fetch completes), and adding a text-search box later is free — it's just another client-side predicate over the same in-memory list, no new query shape needed. This approach only works because the dataset is small (tens of rows, not thousands); it should be revisited (server-side filtering, pagination) if the exercise catalog grows enough that fetching it whole becomes expensive — not applied preemptively while the current numbers make it unnecessary. Exercise cards in the list are already `Pressable` (initially with a placeholder handler, now wired to the detail screen — see below); video (`video_url`) is deliberately not rendered or fetched-for-display anywhere yet — it's carried through `ExerciseRow`/`LocalizedExercise` for completeness but has no consumer until its video-embedding approach (likely `react-native-webview`, not yet added) is built.

## Exercise detail screen and nested Stack inside a native tab

**Context:** `explore` (one of three `NativeTabs` — `expo-router/unstable-native-tabs`, the experimental native-tab-bar API) was a flat file with no push navigation. `NativeTabs` has no JS mock header the way `Tabs` does, so there was no built-in way to push a detail screen or show a back affordance within that tab without extra structure. Separately, `getExerciseById`'s `.single()` throws Supabase's generic `PGRST116` error on a missing row, and the exercise `type` column, while currently constrained by a DB `CHECK` to `'compound'`/`'isolation'`, isn't guaranteed to stay that way from the client's perspective.

**Decision:** `explore.tsx` became `explore/` with a nested `Stack` (`_layout.tsx`, `headerShown: false` to match every other `Stack` in this app) wrapping `index.tsx` (the list, unchanged aside from `handleExercisePress` now calling `router.push`) and `[id].tsx` (new). This is Expo's documented pattern for `NativeTabs` when a tab needs push navigation. The detail screen keeps the app's existing manual-back-link convention (`<Link href="/explore">`, same shape as "Back to sign in" in the auth screens) rather than turning on a native header, for visual consistency with the rest of the app rather than introducing native header chrome in one isolated place. `getExerciseById`'s `.single()` `PGRST116` error is caught specifically and mapped to a distinct not-found state, separate from generic errors — web's equivalent page has no such handling at all. `exerciseRowSchema.type` was loosened from a strict Zod enum to a plain nullable string, and a local `formatExerciseType` in `[id].tsx` maps known values to labels with an explicit default case for anything else — an enum at the schema layer would make the _entire_ fetch throw on an unexpected value, which is the opposite of the graceful handling wanted for that field.

**Consequences:** Any future tab needing push navigation (e.g. a training/programs tab) should follow the same nested-`Stack`-with-`headerShown: false` shape rather than reaching for a native header by default. The detail screen independently re-fetches by id rather than reusing the list's in-memory data (matching web's own behavior) — a deliberate, accepted redundancy at this data size (61 rows), not an oversight; revisit only if a cache layer (TanStack Query) is introduced for other reasons. React Query itself remains deliberately deferred — this is now the third server-state feature (profiles, exercise list, exercise detail) without one, a conscious choice to keep re-evaluating rather than a default.

## Exercise video via YouTube embed

**Context:** `exercises.video_url` is always a YouTube URL, but not a uniform shape — checked live: 57/61 rows are `watch?v={id}`, 4/61 are `shorts/{id}`. Web's detail page embeds video with `data.video.replace("watch?v=", "embed/")`, which does nothing on the 4 `shorts/` rows, leaving a non-embeddable URL as the iframe `src` — a real bug in web, confirmed against actual data, not a hypothetical. `expo-video` can't play YouTube URLs at all (no direct file URL YouTube exposes publicly), so an embedded web view is the only viable approach on mobile too, same underlying reason web uses an iframe instead of `<video>`.

**Decision:** `src/lib/youtube-url.ts` exports `getYouTubeEmbedUrl(url): string | null`, using the `URL` API (hostname, pathname, `searchParams`) rather than a single regex — each supported shape (`watch?v=`, `youtu.be/`, `shorts/`, and already-`embed/`-form URLs) is parsed structurally and always rebuilt into a canonical `https://www.youtube.com/embed/{id}`, fixing the `shorts/` case rather than porting web's bug. `null` in, or anything unparseable/unsupported, returns `null` — no partial/broken URLs ever reach the player. `src/components/youtube-embed.tsx` (native, `react-native-webview`) and `.web.tsx` (a plain `<iframe>`, matching the existing platform-split convention used by `app-tabs.web.tsx`/`animated-icon.web.tsx`) both take only `{ url, title }` — all normalization and the "should anything render" decision stay inside the component, so the detail screen's call site never has to know about embed URLs or YouTube's URL shapes at all. `react-native-webview` needed no config plugin; since this project uses a custom dev client (not bare Expo Go), installing it requires a native rebuild before it's testable.

**Consequences:** Any other embedded third-party player this app adds later (unlikely soon, but the shape is now established) should follow the same self-contained-component pattern — normalization logic lives with the embed, never leaks into call sites. `react-native-webview` has no web-platform support itself, which is exactly why the `.web.tsx` split exists rather than trying to make one component branch internally on `Platform.OS`. No autoplay, no play-state tracking — matches web's actual behavior, not an arbitrary restriction.

## Programs Foundation: dedicated tab, Home reserved for the future Dashboard

**Context:** Training Programs is the next web feature being ported, chosen because web's own exercise-picker component reuses the exercise wiki's data layer and UI directly, and because Complexes/workout-logging/the home feed are all downstream of Programs existing first. The `programs` table has two distinct owner-ish columns — `user_id` (nullable, FK to `auth.users`, what web's own list query filters on) and `author_id` (NOT NULL, FK to `profiles`, used for author-join purposes elsewhere) — easy to conflate, so worth naming explicitly here: the list filters on `user_id`, matching web exactly.

**Decision:** A new, dedicated "Programs" tab (`src/app/(app)/programs/`) rather than repurposing the still-boilerplate Home tab — Home stays untouched, reserved for a future Dashboard once Programs, workout logging, and training history exist to feed it (matching web, where the home feed is itself downstream of those same features). The route is folder-first from day one (`programs/_layout.tsx` + `programs/index.tsx`) rather than starting as a flat file the way `explore.tsx` did — avoiding the flat-file-to-folder migration Explore needed between Sprints 12 and 13 once its detail screen landed. `getPrograms(userId)` (not `getUserPrograms`) to keep the data-layer naming consistent with future sibling functions (`getProgram()`, `getGlobalPrograms()`, etc.). The list-row presentation is a standalone `ProgramCard` component (not inlined in the screen) since web itself reuses the same program-item presentation in more than one place (Programs list, Global Programs) — real, cited reuse, not speculative extraction. `type`/`level` are loosened from the DB's `CHECK`-constrained values to plain strings in the schema, with `ProgramCard` formatting known values and defaulting gracefully otherwise — same reasoning as `exercises.type` in the previous entry.

**Consequences:** Any future tab needing push navigation should default to being created folder-first (`_layout.tsx` + `index.tsx` from the start) rather than repeating Explore's later migration, now that the pattern has been hit twice. The list's "+ Create Program" CTA and each `ProgramCard`'s `onPress` are already wired to placeholder handlers — the create-wizard and detail-screen sprints are pure wiring, not a restructure, matching the exact precedent set by the exercise list before its detail screen existed. Home remaining reserved means there is currently no "landing" content in the app beyond the unmodified Expo Router starter template — that's an accepted, deliberate gap, not an oversight, until the Dashboard's actual dependencies (Programs, logging, history) exist.

## Program Detail: nested query, read-only scope, shared helpers

**Context:** Web's `fetchProgramDetail` loads a program's full tree (days, exercises) in one query with nested PostgREST selects, then a separate call resolves exercise names by id. The same page also embeds workout-logging inputs, read-only training history, edit, and delete — all of which depend on tables/features mobile hasn't built yet (`exercise_drafts`/`exercise_logs`/`training_history`, the creation wizard). Once a second screen needed program type/level formatting and not-found-vs-error detection, `formatProgramType`/`formatProgramLevel` (previously local to `ProgramCard`) and `isNotFoundError` (previously local to `explore/[id].tsx`) became genuine duplicates, not speculative extractions.

**Decision:** `getProgramDetail(id)` in `src/lib/supabase/programs.ts` uses one nested-select query (`programs → program_days → program_exercises`) — the first nested/embedded Postgrest query in this codebase; every other table so far has used flat single-table selects, justified here because a program is genuinely a tree, not a justification for nesting elsewhere by default. `program_days` and `program_exercises` are sorted client-side by `day_number`/`order_index` after the fetch, since embedded relations aren't guaranteed to come back in a deterministic order — a small correctness addition beyond what web itself does (web never explicitly sorts days). Exercise names are resolved via the existing `getExercises()` (already fetches the whole small table) rather than a new by-ids fetch function. `formatProgramType`/`formatProgramLevel` moved from `ProgramCard` into `programs.ts` (now used by both `ProgramCard` and the detail screen), and `isNotFoundError` moved from `explore/[id].tsx` into a new shared `src/lib/supabase/errors.ts` (now used by both detail screens) — both moves triggered by actual duplication appearing, not preemptive extraction. Scope for this sprint is deliberately read-only: title/type/level and the day → exercise list only — no training history, no logging inputs, no edit, no delete, no author link (mobile has no `/profile/[id]` route to link to), no density-mode toggle (web's own modes II/III are visually near-identical, so porting the 3-way toggle wasn't worth it for a first pass). No ownership check either — nothing on this screen is ownership-gated yet, unlike web's `isMyProgram` gate, which only matters once edit/delete/logging exist.

**Consequences:** Any future table with a genuine tree/hierarchy shape (not just a one-to-many list) is a reasonable candidate for the same nested-select pattern; flat tables should keep using flat selects. `isNotFoundError`/`errors.ts` is now the place any future detail screen's `.single()` not-found handling should import from, not reimplement. This sprint deliberately doesn't distinguish "viewing your own program" from "viewing someone else's" — that distinction was introduced in Sprint 21 alongside Delete (see "Program Deletion" below); Edit and logging still need to add their own owner-only affordances on top of the same check when they land.

## Zustand

**Context:** Need a place for client-only/UI state (e.g. active tab, in-progress form drafts, ephemeral UI flags) that doesn't belong in the server-state cache.

**Decision:** Zustand for client state — minimal API, no boilerplate, no providers required.

**Consequences:** State that mirrors the backend goes through TanStack Query, not Zustand. Zustand stores stay small and UI-focused; if a store starts caching server data, that's a sign it should be a query instead.

## React Hook Form

**Context:** Auth, onboarding, and profile screens need validated forms with reasonable performance (no full-tree re-render per keystroke).

**Decision:** `react-hook-form` for form state and validation wiring, paired with Zod resolvers.

**Consequences:** Form validation schemas are defined once in Zod and reused for the resolver — no separate hand-written validation logic per form.

## Zod

**Context:** Validation is needed in three places: parsing environment variables, validating form input, and validating/typing Supabase responses. Using a different tool for each would triplicate effort.

**Decision:** Zod as the single schema/validation library across all three use cases.

**Consequences:** One schema can, in principle, back both a form's validation and the shape of the data it submits to Supabase. Environment variable parsing (when added) uses Zod schemas rather than manual `if` checks.

## Password recovery as a dedicated auth state

**Context:** Supabase's password-recovery flow (`resetPasswordForEmail` → emailed link → `exchangeCodeForSession`) establishes a real session, indistinguishable from a normal login by session shape alone. The app's root navigator gates access to `(app)` purely on session presence, so without further care a recovery session would satisfy that guard and drop the user straight into the main app before they've actually changed their password.

**Decision:** Track password recovery as its own `AuthStatus` value (`passwordRecovery`), set only in response to Supabase's authoritative `PASSWORD_RECOVERY` auth event — never inferred from session presence alone, and never treated as equivalent to `authenticated`. `reset-password` is gated behind `Stack.Protected guard={status === 'passwordRecovery'}`, structurally separate from the `authenticated` guard on `(app)`, so a recovery session can reach only the reset-password screen and nothing else in the app. After a successful `updateUser({ password })`, the recovery session is explicitly ended (`signOut()`) and the user is redirected to `/login` (with a success flag) rather than into `(app)` — password reset always ends in an explicit, intentional sign-in with the new credential, not an implicit continuation of the recovery session.

**Consequences:** The auth state machine stays clean and closed: `passwordRecovery` is only ever entered via the real Supabase event and only ever exited via sign-out, so it can't be spoofed by a malformed deep link or silently upgraded to a full session. This does mean password reset costs the user one extra sign-in step compared to auto-login after reset — accepted as the safer default. The platform-specific parts of the flow (deep-link URL construction, reading `code`/`type` from the callback route) are isolated in `src/lib/supabase/auth.ts` and `auth-callback.tsx`; the state machine itself, the Supabase call shapes (`resetPasswordForEmail`, `updateUser`), and the redirect-to-login behavior are platform-agnostic and intended to carry over to the future web implementation with only those isolated parts needing a web-specific rewrite (browser redirect URL instead of a deep link, `detectSessionInUrl`-based session capture instead of manual code exchange).

## StyleSheet + theme tokens

**Context:** Need a consistent visual language (colors, spacing) across components without committing early to a heavier styling library.

**Decision:** Keep React Native's built-in `StyleSheet` API, backed by the existing token system in `src/constants/theme.ts` (`Colors`, `Spacing`, `Fonts`) and the `useTheme` hook. `theme.ts` already documents NativeWind, Tamagui, and Unistyles as known alternatives.

**Consequences:** Styling stays dependency-free and close to React Native primitives. This is deliberately deferred, not rejected outright — revisit only if the current approach causes real friction as the component set grows (see `docs/roadmap.md`).

## Program Creation Wizard: shared store instead of local state, deferred write

**Context:** Web's creation wizard collects all 5 steps (name, type, level, days count, then a per-day exercise picker) in the parent component's local React state and writes to Supabase immediately after the final step — three sequential, non-transactional inserts (`programs` → `program_days` → `program_exercises`) with a real, confirmed orphan-row risk on partial failure, and (separately) an edit flow that's actually broken in web today: `updateTrainingProgram` exists but is never called, so editing an existing program silently creates a duplicate rather than updating it. Mobile's wizard is being built incrementally across sprints, and Sprint 19's exercise picker is expected to be a separate route (reusing the exercise wiki's browsing UI), not a step embedded in the same screen — a pushed route has no clean way to hand data back into a specific still-mounted parent screen's local state; route params only flow parent → child.

**Decision:** Wizard data (name, type, level, days count — exercises to be added when Sprint 19 needs them) lives in a temporary Zustand store (`src/stores/program-wizard-store.ts`), not local component state — exactly the "in-progress form drafts" use case this project's own Zustand decision already anticipated, not a new pattern. The current step number stays local `useState` in the wizard screen, since it's pure UI/navigation state, not data that needs to survive a screen swap. The store resets on three triggers only — starting a fresh Create flow, an explicit Cancel, and (in the later sprint that adds it) a successful final Create — deliberately _not_ on navigating between wizard steps or returning from the future exercise picker, which Expo Router's `Stack` supports naturally since it keeps the wizard screen mounted underneath any screen pushed on top of it. Sprint 18 itself performs **no Supabase writes at all** — steps 1–4 only collect data locally; the single write (`programs` + `program_days`, and later `program_exercises`) is deferred until exercise selection is also complete, specifically to avoid ever creating a partially-empty program row, which is a stronger guarantee than web provides today.

**Consequences:** Any future multi-screen flow with the same "a later screen needs to hand data back to an earlier one" shape should default to a small, scoped Zustand store rather than route params or React Context — Context would be a second, redundant pattern for a problem this app already has an established solution for. The final write landed in Sprint 20 (see "Program Creation Wizard: final write" below), keeping the same sequential-insert approach web uses rather than introducing a transaction, but with a narrower blast radius than web's (nothing is written until the entire flow, including exercises, is ready) and with app-layer cleanup of already-committed rows on partial failure, which web itself doesn't have. Building true edit/update semantics later still means writing that logic from scratch — web's own `updateTrainingProgram`, while more complete than what's wired up, is unused/undertested and has a real data-shape bug (it expects `program_exercises.id` values but the wizard state actually holds raw `exercise_id`s), so it should be treated as reference material at most, not a source to port directly.

## Exercise Picker: pushed route, shared browsing logic, scoped Confirm/Cancel

**Context:** Web's exercise picker is a per-day modal (full-screen CSS overlay) with its own muscle-group tabs and, critically, no search box at all — mobile's Explore screen already has both search and muscle-group filtering built (Sprints 12–16), and the picker needs the exact same browsing behavior, just with tap-to-select instead of tap-to-navigate. Building the picker meant either duplicating Explore's fetch/filter logic and search/filter UI, or extracting it — with a second real consumer now needing the identical logic, this crossed from "one screen's implementation detail" into "actual duplication," the same bar this project has applied consistently (`formatProgramType`/`isNotFoundError` were extracted the same way, once a second consumer appeared, not before).

**Decision:** `src/hooks/use-exercise-browser.ts` now holds the fetch/filter logic that used to live directly in `explore/index.tsx` (unchanged in shape — same `loadState`, same client-side muscle-group + search filtering) — Explore was rewritten to consume it rather than the picker copying Explore's code. `src/components/exercise-search-bar.tsx` and `src/components/muscle-group-filter.tsx` are the same extraction for the search input row and the filter-pill row, both now shared between Explore and the picker. The picker itself (`src/app/(app)/programs/exercise-picker.tsx`) is a pushed `Stack.Screen` registered as a new sibling in the existing `programs/_layout.tsx` — no modal presentation, matching this app's zero existing precedent for it and avoiding the accidental-dismiss risk a swipeable modal would introduce for a screen whose whole point is a deliberate Confirm-vs-Cancel choice. Selection state is local `useState` in the picker, seeded once from `program-wizard-store.ts` on mount and written back via `setDayExercises` only on Confirm — Cancel calls nothing but `router.back()`, touching the store not at all, mirroring web's own Cancel/Confirm boundary exactly (the one piece of web's design kept as-is). The wizard's step 4 now renders one card per day (`Array.from({ length: daysCount })`, reading `days[dayIndex]` from a single whole-array store subscription rather than one hook call per day inside the loop, which would violate the Rules of Hooks), each with its own "Add/Edit exercises" action passing that day's real index as a route param — replacing the earlier hardcoded `dayIndex=0` placeholder from the first implementation phase.

**Consequences:** Any future screen needing the exercise catalog with search/filtering should consume `useExerciseBrowser` rather than re-fetching — there are now two real call sites proving the extraction was warranted, not a guess. The picker's selection UI (numbered badges via array position, `theme.backgroundSelected` for the selected-card background) intentionally reuses this app's own established "selected" visual language rather than porting web's opacity-based approach, and shows a running "Confirm (N)" count web itself lacks — a small, deliberate improvement, not an oversight. The final wizard-submit button (`handleFinishPress` at the time) was still a placeholder as of this sprint — this sprint only made the full local-state flow (name → type → level → days → per-day exercises) work end to end; the actual write was wired up in Sprint 20 (see "Program Creation Wizard: final write" below), which also renamed the handler to `handleCreatePress`.

## Program Creation Wizard: final write — sequential inserts kept, with best-effort cleanup

**Context:** Sprint 18 deferred the final write and flagged the sequential-insert, no-transaction risk inherited from web as accepted-but-not-solved, noting no Postgres RPC/transaction existed anywhere in this project. Landing the actual write meant deciding explicitly, for the first time in this project, between two strategies: match web's three sequential inserts (`programs` → `program_days` → `program_exercises`) or introduce this project's first Postgres RPC/transaction to make the write atomic. Web's own insert order isn't just a data dependency (each step needs the previous step's generated ids) — it's required by the live RLS INSERT policies on `program_days`/`program_exercises`, both of which authorize via an `EXISTS` subquery through the parent row(s), confirmed directly against the Supabase project rather than assumed.

**Decision:** Sequential client-side inserts were kept (`createProgram` in `src/lib/supabase/programs.ts`), not moved to an RPC. Introducing this project's first SQL migration/RPC tooling — no `supabase/migrations` directory or authored SQL exists anywhere in this repo — to solve one feature's atomicity was judged disproportionate, consistent with this project's repeated pattern of deferring new architectural surface until a real, not speculative, need exists (the same standard applied to TanStack Query every time it's come up). The orphan-row risk this inherits from web is mitigated, not eliminated, at the application layer: on a `program_days`/`program_exercises` failure, `cleanupOrphanedProgram` deletes whatever was already committed before re-throwing the original error, rather than leaving a partially-created program behind. This mitigation is simpler than it first appeared — live inspection of the foreign-key constraints (`program_days.program_id → programs.id` and `program_exercises.day_id → program_days.id`, both confirmed `ON DELETE CASCADE` via a direct `pg_constraint` query against the linked project, not inferred) confirmed that cascade delete makes explicit `program_exercises` cleanup unnecessary: deleting `program_days` alone cascades through any `program_exercises` rows underneath it, so cleanup only needs two explicit deletes (`program_days`, then `programs`), not three. Web's own `deleteProgramWithRelations` deletes all three tables explicitly, which is safe but redundant, not a sign that cascade delete is absent.

The wizard integration (`src/app/(app)/programs/create.tsx`) also settled three UX questions: the Create button (and Cancel, and the step-4 in-page Back link) are disabled while the write is in flight, using the same `isSubmitting`-gated pattern already used by `reset-password.tsx`/`forgot-password.tsx`; success resets the wizard store and then navigates via `router.replace` (not `push`) to the new Program Detail screen — an improvement over web, which always redirects to a list — with `replace` chosen specifically so back-navigation from the new program doesn't land on a stale, reset wizard; failure leaves all wizard data untouched and re-enables the same Create button as the retry path, with the error rendered via this app's existing `❌ {message}` + `themeColor="danger"` pattern already used in `programs/[id].tsx` and `exercise-picker.tsx`.

**Consequences:** This decision should be revisited, not silently carried forward, once Edit or Delete implementation begins — both have messier failure modes than Create (Edit's positional diff can leave old and new data commingled on a partial failure; a future `delete_program` RPC would remove the reverse-order manual delete entirely) and may tip the balance toward an RPC for reasons specific to those features, not just because the sequential pattern is already in place here. Any future multi-table write in this project should default to the same pattern established here — sequential inserts plus app-layer compensating cleanup on failure — until a genuine transactional need, not just tidiness, justifies introducing this project's first RPC/migrations workflow.

## Program Deletion: single cascading delete, ownership gate, local platform branch

**Context:** Web's `deleteProgramWithRelations` manually deletes `training_history` → `program_exercises` → `program_days` → `programs`, in that order, mirroring the reverse of its own create function's insert order. Sprint 20 had already confirmed `program_days.program_id → programs.id`, `program_exercises.day_id → program_days.id`, and `training_history.day_id → program_days.id` are all `ON DELETE CASCADE`. A fresh check for Sprint 21 (`pg_policies`/`pg_class.relrowsecurity` against the linked project) additionally found `program_days`/`program_exercises` have RLS enabled with only `SELECT`/`INSERT` policies — no `UPDATE` or `DELETE` policy exists on either table. That means web's own explicit `program_exercises`/`program_days` deletes are themselves silent RLS no-ops today (0 rows affected, no error); its function only actually works end-to-end because Postgres executes `ON DELETE CASCADE` actions as the table owner, which bypasses RLS (confirmed `relforcerowsecurity = false` on both tables) — the final `programs` delete's cascade is doing all the real work regardless of what came before it. Program Detail (Sprint 17) also had no ownership check at all, deliberately deferred until an owner-only affordance actually existed.

**Decision:** `deleteProgram(id)` in `src/lib/supabase/programs.ts` is a single `supabase.from('programs').delete().eq('id', id)` call — not a port of web's four-step version, since three of those four steps are either redundant (cascade already covers them) or non-functional under the live RLS policies as they stand today. `ProgramDetailRow`/`PROGRAM_DETAIL_QUERY` gained `user_id` (nullable, matching the live column) specifically to support an ownership check: `programs/[id].tsx` renders the Delete action only when `program.user_id === user?.id` (`useAuthStore`), the same comparison web's `isMyProgram` makes. Confirmation uses an inline `Platform.OS === 'web'` branch directly in `programs/[id].tsx` — `Alert.alert` on native, `window.confirm` on web (verified necessary by reading `react-native-web`'s actual installed source: `Alert.alert()` is a literal no-op there, not assumed). This stays inline rather than becoming a shared `src/lib/` module: no second real consumer exists today — Edit isn't a destructive action, and a Global Programs "remove" confirm is speculative and out of current roadmap scope — matching this project's standing rule to extract only once a second real consumer exists, not because a platform split "might" be reused later. On success, `router.replace('/programs')` — not `push` or `back()` — verified against the exact v57 Expo Router docs that `replace` removes the current screen from navigation history entirely, so back-navigation cannot return to the just-deleted Program Detail screen; `replace` was chosen over `back()` specifically because `back()`'s correctness would depend on assuming this screen was always reached via a single push from the list, an assumption `replace`'s explicit destination doesn't need.

**Consequences:** Any future destructive-confirm need, if and when one actually appears, should default to the same inline `Platform.OS` branch unless a second real consumer makes extraction concretely worthwhile — this is now the explicit precedent for that call, not just an implicit norm. The RLS gap found here (`program_days`/`program_exercises` missing `UPDATE`/`DELETE` policies) is a hard blocker for Edit, not just a Delete concern — Edit cannot port web's `updateTrainingProgram` as-is, since its `.update()`/`.upsert()` calls on `program_exercises` would be silent no-ops under the same policies today; this needs its own explicit decision (new RLS policies vs. some other approach) before Edit is implemented. Program Detail's ownership check (`user_id === user?.id`) is now established and should be reused as-is when Edit adds its own owner-only affordance, rather than reimplemented.

## Google Sign-In and Register parity: browser-redirect OAuth, functional-only scope

**Context:** Web already has working Google Sign-In (`supabase.auth.signInWithOAuth`) and collects `firstName`/`lastName` at registration, populating `full_name`/`avatar_url` user metadata; mobile had neither. A full audit compared every auth screen against web in detail (colors, buttons, inputs, spacing) and found mobile's visual language is a different, self-consistent system already used across every other screen (Programs, Profile, Explore) — not a variant of web's. Bringing auth screens to full visual parity would mean either diverging mobile's own auth screens from the rest of the app, or updating `theme.ts`/shared button and input styling app-wide, both far larger than an authentication sprint.

**Decision:** Scope was deliberately narrowed to functional parity only — Google Sign-In and the register metadata gap — with `theme.ts`, `AuthTextField`, shared button styling, typography, and colors explicitly untouched. For Google Sign-In, the browser-redirect strategy was chosen over the native `@react-native-google-signin/google-signin` SDK (Supabase's generic top recommendation): `signInWithGoogle` (`src/lib/supabase/auth.ts`) uses `expo-web-browser` (already a dependency since the project's first commit, previously unused, so no new native rebuild was needed) via `WebBrowser.openAuthSessionAsync`, reusing the existing `authCallbackUrl()`/`exchangeCodeForSession` PKCE plumbing built for password reset and email confirmation — consistent with this project's standing pattern of avoiding new native dependencies/config until a simpler path is proven insufficient. It does **not** route through `/auth-callback` on native — `openAuthSessionAsync` captures the redirect directly in the calling code, unlike the email-triggered flows that rely on a genuine OS-dispatched deep link. Re-checking Supabase's current docs before implementing (rather than relying on the initial research) surfaced one addition the original plan missed: on the web target, the same `/auth-callback` route needs a web-only, module-scope `WebBrowser.maybeCompleteAuthSession()` call so a Google sign-in opened as a popup can signal completion back to its opener — confirmed unsupported (not just inert) on native, so it's guarded by `Platform.OS === 'web'`. The Google icon reuses web's exact official SVG asset (`assets/images/google-icon.svg`) rather than a rasterized fallback — confirmed both that Metro's default `assetExts` already includes `svg` and that `expo-image` (already used throughout the app) renders SVG natively on iOS/Android/web, so no new dependency (`react-native-svg`) was needed. Register's `firstName`/`lastName` fields and their validation mirror web's actual `firstNameOptions`/`lastNameOptions` (`lib/validations.tsx`), and `signUpWithPassword` now builds `full_name`/`avatar_url` metadata identically to web's `registerUserWithEmail` — including reusing the same third-party `ui-avatars.com` generator web relies on, since web has no shared helper for this either, just an inline expression mobile now mirrors rather than abstracts.

Manual verification on web surfaced a real double-exchange race not caught by initial testing: `openAuthSessionAsync` has no OS-level session capture on web (unlike `ASWebAuthenticationSession`/Custom Tabs on native), so it opens a real popup that genuinely navigates to and loads the actual `/auth-callback` route — the same screen password reset and email confirmation use, which already unconditionally called `exchangeCodeForSession` on its own. That raced against `signInWithGoogle()`'s own exchange in the opener window; whichever attempt was still in flight when the opener's `dismissPopup()` closed the popup failed with a raw `TypeError: Failed to fetch` (the request's execution context torn down mid-flight, not a clean HTTP rejection from Supabase). Fixed by capturing `maybeCompleteAuthSession()`'s return value (previously called and discarded) and skipping the popup's own exchange whenever it reports `'success'` — that result is only ever `'success'` when this load genuinely is the OAuth popup, in which case the opener is already about to perform the one real exchange. This has no effect on native (the value is always `null` there, guarded by the same `Platform.OS === 'web'` check) or on password reset/email confirmation on web (`maybeCompleteAuthSession` reports `'failed'` for a direct page load, so their existing exchange behavior is unchanged).

**Consequences:** Manually verified working end-to-end on the custom development build and web, including the double-exchange fix above — this is this project's first OAuth flow of any kind, and the browser-redirect pattern (not the native SDK) is now the established precedent for any future provider, unless a concrete need for native-SDK-quality UX arises later. Two prerequisites were required outside this repo and confirmed necessary but not independently checkable by any tooling available during implementation: the mobile redirect URL present in Supabase's dashboard redirect-URL allow-list, and Supabase's fixed callback URL registered in Google Cloud Console — both are external configuration this repo cannot track or verify, only rely on. Any future screen that both (a) opens a popup-based web flow via `openAuthSessionAsync` and (b) redirects back to a route with its own independent side effects should check for this same class of race, not assume a single call site. Full visual/design-system parity with web (colors, button/input styling, typography) remains explicitly deferred, tracked in `docs/roadmap.md` as its own future cross-app redesign, not folded into this sprint.

## Shared screen layout/header: component-based, not Stack-level

**Context:** Web's persistent `Header`/`AppShell` wraps every authenticated page; mobile has never had an equivalent — each screen manages its own top-of-screen back link/title ad hoc, duplicated near-verbatim across most of them (the `SafeAreaView` + centered themed container shell repeats across every auth screen; the back-link `Link`/`ThemedText` pair repeats across every detail screen). Expo Router offers two genuinely different ways to introduce shared chrome: a Stack-level `screenOptions.header` (automatic for every screen under that Stack) or a plain component each screen renders in its own body. This project has already deliberately chosen `headerShown: false` everywhere, specifically "for visual consistency with the rest of the app" (see the Exercise detail screen entry above) — reversing that would mean re-litigating a decision already made, not just adding a component.

**Decision:** Two separate, independently-adoptable components — `ScreenLayout` (`src/components/screen-layout.tsx`, the shared shell) and `ScreenHeader` (`src/components/screen-header.tsx`, a back/title/right three-slot row) — kept as plain screen-rendered components, not Stack header options. This needed zero changes to any `_layout.tsx`: `NativeTabs` still has no header slot, both existing nested Stacks (`explore/_layout.tsx`, `programs/_layout.tsx`) keep `headerShown: false` untouched, and dynamic titles (Exercise Picker's day number) are just a prop from state already in hand, not `navigation.setOptions()` indirection. `ScreenHeader`'s back affordance supports both shapes already in use across the app — a static `backHref` (a `Link`) and an `onBackPress` handler (a `Pressable`, for cases needing logic before navigating) — and deliberately doesn't standardize the exact wording ("← Back to programs" vs "Cancel"), leaving that to each call site, since it already varied before this component existed and shouldn't silently change as a side effect of the refactor. The title slot uses `smallBold`, matching Exercise Picker's pre-existing compact header-title size, not the app's 48px hero `title` style — screens with a real page-hero title (Program Detail's program name) keep rendering it separately, below the header. Both components were adopted immediately rather than introduced unused: `ScreenHeader` in `programs/[id].tsx`, `explore/[id].tsx`, and `programs/exercise-picker.tsx`; `ScreenLayout` in `confirm-email.tsx` (its container style is an exact match, copied directly from it). The Create Wizard's Cancel (store-reset-then-navigate) and the remaining auth screens (`login.tsx`, `register.tsx`, `forgot-password.tsx`, `reset-password.tsx`) were deliberately left for a later pass — the wizard's Cancel has extra semantics worth handling once the component's shape is proven elsewhere first, and the remaining auth screens' back links sit in a footer, not the top row `ScreenHeader` targets.

Separately, `app-tabs.web.tsx`'s tab bar chrome still showed "Expo Starter" branding and a `docs.expo.dev` link — leftover, never-touched Expo Router starter-template boilerplate (only the tab `href`s had ever been updated to this app's real routes). Replaced with the app's actual name; no logo image was introduced, since web's own logo asset (`public/logos/logo.png`) turned out to be a large glow/hero graphic, not sized for inline header use, and mobile's own app icon is still Expo's unmodified default — real logo/icon artwork is a separate, unstarted concern, not a decision made here.

**Consequences:** Any future screen needing a back/title header row should default to `ScreenHeader` rather than hand-rolling another `Link`+`ThemedText` pair; any future screen with the plain centered-shell shape should default to `ScreenLayout`. Adopting `ScreenHeader` changed Exercise Picker's title from right-aligned (in a two-child `space-between` row) to horizontally centered (`ScreenHeader`'s title slot is `flex: 1, textAlign: 'center'`) — a small, deliberate visual change, not an oversight. Neither component needed a `.web.tsx` split — both are pure composition of already-cross-platform primitives, no platform-specific behavior involved. Real branded logo/icon artwork remains an open, unstarted gap, tracked in `docs/roadmap.md`.

## Design system alignment with web: outlined buttons/inputs/cards, phase one

**Context:** A full audit (colors, typography, radius, shadows, buttons, cards, inputs) confirmed web has one cohesive visual language — navy-tinted backgrounds/text, a shared `--border-color`, 4px radius, no shadows, no bright accent color — applied uniformly via borders, not fills, across buttons/inputs/cards alike; mobile had a different but equally cohesive system (pure black/white, filled `#3c87f7` buttons hardcoded across 8+ files, 8px radius, filled `backgroundElement` cards/inputs). Sprint 22 had explicitly deferred closing this gap as "its own future cross-app redesign, not an auth-specific change," specifically to avoid the exact risk of a large, all-at-once visual refactor.

**Decision:** `theme.ts` gained web's actual palette values (confirmed live against `ui/variables.scss`, not approximated): `text`/`background` updated, new `title`/`border` tokens added. `backgroundElement`, `backgroundSelected`, `textSecondary`, and `danger` were deliberately left unchanged — web has no distinct "element background" concept (its cards/inputs are bordered against the page background, not filled with a separate shade), and changing those values would have rippled into Explore/muscle-group-filter/exercise-picker selection styling, none of which this phase touches. `ThemedText`'s `linkPrimary` had its own hardcoded `color: '#3c87f7'` removed so it falls through to the theme's normal `text` color resolution — web has no special "link blue" either, just colored text, occasionally underlined.

A new shared `Button` (`src/components/button.tsx`) replaced the 8+ duplicated inline `Pressable` + hardcoded-style blocks: outlined (`transparent` fill, `2px` border in the new `border` token, `Spacing.one` radius, `minHeight: 40`), rendering `children` as-is rather than forcing a text wrapper, so migrated call sites kept their own `ThemedText`/icon+text content unchanged. It also accepts an optional `style` prop — a deliberate, narrow escape hatch (not a general styling API) used only by the Program Creation Wizard's step-nav buttons to preserve their pre-existing extra horizontal padding, rather than silently losing that sizing during the migration. Only primary filled-action call sites were migrated (`login-form.tsx`, `register-form.tsx`, `confirm-email.tsx`, `forgot-password.tsx`, `reset-password.tsx`, `programs/index.tsx`, `programs/create.tsx`, `programs/exercise-picker.tsx`); destructive/quiet/text-only actions (Delete Program, Cancel/Back/Retry links, the wizard's per-day "Add/Edit exercises" links) were deliberately left as plain themed text, since `Button` is meant for one role, not every tappable-text pattern in the app.

`AuthTextField` switched from a filled `backgroundElement` box to the same bordered treatment (`transparent` fill, `2px` border, switching to `theme.danger` when an error is present — matching web's border-turns-red error signal — `Spacing.one` radius, `minHeight: 40`), and its error text switched from a hardcoded `color: '#e5484d'` to `themeColor="danger"` while that line was already being touched. `ProgramCard` dropped its `backgroundElement` fill (now inherits the page background) in favor of a `1px` border (web's own cards use a thinner border than its buttons/inputs, `1px` vs `2px` — mirrored here rather than flattened to one width) and `Spacing.one` radius, while **deliberately keeping its existing left-aligned title/meta/chevron row** rather than adopting web's centered text — a specific decision confirmed directly rather than assumed, since centered text read as web-specific rather than core to the bordered-design-language parity being adopted. `ScreenHeader` needed almost no direct change — it hardcodes no colors of its own, only `ThemedText` types, so it inherits the new palette automatically; the one direct tweak was widening its internal row gap to match web's more generous header spacing.

Explicitly out of scope for this phase, not forgotten: `NativeTabs`/`app-tabs.web.tsx` structure (icons + bottom placement on native stay exactly as they are — matching this project's platform-appropriate-over-literal-parity precedent already set when `NativeTabs` was first chosen), pixel-identical matching of any specific web measurement, and the remaining `backgroundElement`/`backgroundSelected`-based screens (Explore, muscle-group filters, exercise-picker selection, which still has one remaining hardcoded `#3c87f7` badge).

**Consequences:** Any future primary action should default to `Button` rather than a new hand-rolled `Pressable`+style block — this is now the single place that styling lives. The tab bar (`app-tabs.tsx`/`app-tabs.web.tsx`) already reads `Colors[scheme]` directly for its own colors, so it inherited this palette change automatically with zero code touched there, consistent with "structure stays native, only the color it already reads changes." Phase two (Explore/muscle-group-filter/exercise-picker selection styling, the remaining hardcoded badge color, and any native tab bar color re-tuning) is tracked separately in `docs/roadmap.md`, deliberately not bundled into this phase, for the same "keep it incremental, prove it in production before expanding" reasoning already applied throughout this project's history.

## Home tab: global training-history feed, ported as-is

**Context:** Web's Home page is literally one component — a global, cross-user activity feed of every completed workout entry, unfiltered by current user, no pagination, no personalization. Mobile's Home tab was still 100% untouched Expo Router starter-template boilerplate. Sprint 25's own research phase recommended building a personalized programs summary in that slot instead (reusing `getPrograms`/`ProgramCard`, already built), given the feed shows *other* users' data and mobile has no way to contribute to it yet (Workout Logging doesn't exist). That recommendation was explicitly overridden: the direction given was to port the feed web actually has, not substitute a different concept, so this entry documents what was built, not the alternative that was proposed and declined.

**Decision:** `src/lib/supabase/training-history.ts` (new) holds `getTrainingHistoryFeed()`, joining `training_history` → `profiles` via the single live-confirmed FK (`fk_user`) — no ownership filter, matching web's `fetchAllTrainingHistories` exactly. `getExerciseIdsForProgramExercises` was added to `programs.ts` (co-located with the `program_exercises` table it queries, matching this project's one-file-per-table convention) to resolve `training_history.values`' keys (`program_exercise_id`s) down to real exercise names via the already-existing `getExercises()`/`localizeExercise`, mirroring web's two-step `fetchProgramExerciseMap` + `fetchExercisesByIds`.

Two correctness improvements over web's own code, found by checking the live schema rather than assuming it matches web's implicit assumptions: `training_history.user_id` is nullable (0 of 14 current rows are, but the column allows it), so `profiles` is modeled as nullable in the Zod schema and the UI falls back to "Unknown" — web's own code has no such guard and would break on a row like that. Sorting was moved server-side: chained `.order('date', {ascending:false}).order('created_at', {ascending:false})` rather than porting web's client-side `.sort()` comparator — confirmed via the installed `@supabase/postgrest-js` source (each `.order()` call appends to the request's order parameter rather than overwriting it) that this produces the exact same `ORDER BY date DESC, created_at DESC` semantics as web's two-level comparator, not assumed equivalent.

Two deliberate, smaller adaptations: dates use `toLocaleDateString(undefined, ...)` (device locale), matching the convention `profile.tsx` already established, rather than porting web's hardcoded `'uk-UA'`, which isn't part of this app's actual i18n system and wouldn't match anything else already built. The feed's avatar/username renders as plain, non-interactive info rather than web's link to `/profile/{userId}` — mobile has no route to view another user's profile at all, and building one is a distinct feature beyond "port the feed," not folded in here.

One implementation gotcha worth recording for future similar code: an initial version mixed a non-promise value into `Promise.all([value, promiseA, promiseB])`, which caused TypeScript to lose positional tuple inference entirely and union all three result types together — not a runtime bug, but a confusing type error. Restructured as a plain sequential `.then()` chain instead, which avoided the issue entirely; future multi-fetch loading code in this app should default to that shape rather than mixing already-resolved values into `Promise.all`.

**Consequences:** The feed can only ever grow via web today — mobile has no logging feature yet to post new entries into it, so this is read-only parity, not full parity, until Workout Logging lands. The avatar-with-fallback-initial UI now exists in two places (`profile.tsx` and this feed) with no shared component between them; not extracted here since that would mean touching `profile.tsx`, outside this sprint's stated scope — a small, real duplication a future sprint could address once (if) a third consumer makes the case clearer, matching this project's standing extraction bar. The `/profile/[id]` gap is now tracked explicitly in `docs/roadmap.md` rather than left implicit.
